# AutoLava AI 易用性与可扩展基础升级设计

> 状态：已讨论并批准
> 日期：2026-07-15
> 适用分支：`feature/phase-1-foundation`
> 设计目标：在不改变首页核心内容的前提下，修复一期系统的实际易用性问题，引入可配置且可追溯的收入组成，并为人员管理、AI 与自动化阶段建立稳定扩展边界。

## 1. 背景与现状

本设计建立在一期系统的真实浏览器走查和原始设计、实施计划复核之上。当前系统已经具备登录、店铺权限、管理员 CRUD、每日经营记录、天气回退、覆盖与审计、删除与回滚、筛选、导出、经营简报、图表、本地启动和 CI 等核心能力。

实际使用暴露出以下问题：

- 首页遗漏原设计中的“立即记账”。
- 手机底部导航包含六个入口，发生越界和内容遮挡。
- 店铺选择没有可靠持久化，刷新或直接进入页面后需要重复选择。
- 记账桌面页面过于松散，手机保存与导航关系不清晰。
- 手机记录表格需要横向滚动，编辑与删除入口难以触达。
- 桌面记录页同时展开大量记录与审计事件，页面可能达到数千像素高度。
- 数据表格遗漏记录人和最后修改人。
- 图表缺少快捷日期范围，单一类别饼图没有分析意义。
- 管理功能集中在单一六标签页面，后续扩展困难。
- 当前管理员可以在界面上停用自己，后端缺少最后管理员保护。
- 响应式测试只验证容器没有根级横向溢出，未覆盖底部遮挡和真实操作可达性。
- 测试店铺数据会出现在管理员的正常店铺列表。
- Sulmona 已导入数据中的活动字段包含误填内容，收入类别名称也与“系统只记录收入”的业务口径冲突。

## 2. 已确认的产品原则

1. 系统只记录收入，不记录利润或费用。
2. 手机端服务家庭成员的高频日常操作，桌面端服务店主的管理与分析。
3. 首页保持原始结构：昨天、今天、明天三张卡片，下面是“立即记账”；不得擅自加入其他经营指标卡片。
4. 店铺未配置收入组成时，用户直接填写总收入。
5. 店铺启用收入组成时，用户只能填写组成项，总收入由管理员指定的组成项相加得出，用户不能直接修改总收入。
6. 组成项可以仅记录而不计入总收入。
7. 历史总额记录不强制拆分；配置变化不反向改变历史数据。
8. 不开发 Excel 导入界面。一次性数据导入继续使用受控脚本。
9. 本次采用“基础升级 + 易用性改造”，不提前实现人员、AI 和自动化业务，但所有边界必须允许后续模块自然接入。

## 3. 信息架构与导航

### 3.1 手机端

底部导航固定为四个入口：

- 首页
- 记账
- 记录
- 更多

图表、管理中心、账号信息和退出登录放入“更多”。底部导航必须占用明确的布局安全区，所有页面的最后一个可操作元素都能滚动到导航上方，不允许通过覆盖内容实现固定导航。

### 3.2 桌面端

使用固定侧边栏：

- 首页
- 每日记账
- 记录数据库
- 图表分析
- 管理中心

当前店铺切换和登录账号常驻显示。管理中心内部按“店铺、用户与权限、收入组成、系统设置、审计记录”分组，不再把所有管理能力平铺为单页标签。

侧边栏使用模块注册方式生成。后续人员管理、AI 助手和自动化中心通过新增模块与权限声明接入，不要求重写应用外壳。

### 3.3 店铺上下文

用户选择店铺后，将选择保存在本地会话状态；重新加载、直接访问子页面或再次打开系统时恢复最近有权访问的店铺。若权限已被撤销，自动回退到第一个有权限的店铺，并清除失效选择。

## 4. 页面与交互设计

### 4.1 登录

- 保持账号密码登录。
- 提供明确的加载、失败和账号停用提示。
- 未登录时的认证检查不应在正常用户界面产生干扰性错误。
- 补齐 favicon、页面标题和基础品牌信息。

### 4.2 首页

- 内容严格保持“昨天 / 今天 / 明天”三张卡片。
- 卡片展示原设计已有的经营、记账或天气信息，不扩展为月度 KPI 看板。
- 三张卡片下方提供醒目的“立即记账”，直接进入当前店铺的记账页。
- 无数据、天气失败和当日已记账分别提供明确状态。

### 4.3 每日记账

采用紧凑单页：

- 日期和营业状态直接显示。
- 收入区域直接显示。
- 天气优先自动获取，失败时允许手动补充或稍后刷新，不能阻止保存。
- 洗车、活动和备注属于低频字段，默认折叠，展开后保留用户输入。
- 手机保存按钮固定在底部导航安全区上方。
- 保存成功后提供明确反馈，并允许进入刚保存记录的详情。

收入区域有两种互斥模式：

**总额模式**

- 店铺没有启用收入组成。
- 用户填写当日总收入。
- 记录标记为 `legacy_total` 或等价的总额模式。

**组成模式**

- 显示当前配置版本中的所有启用组成项。
- 用户填写各组成金额。
- 总收入只读，并实时显示被标记为“计入总收入”的项目之和。
- 后端重新计算并验证总收入，不能信任客户端传入的合计。
- 不计入总收入的组成仍被保存和展示。

### 4.4 手机记录页

使用紧凑表格，只显示：

- 日期
- 总收入
- 营业状态
- 详情入口

不得产生页面级横向滚动。详情页展示天气、洗车、活动、备注、收入组成、记录人、最后修改人及审计历史，并提供有权限的编辑、删除和回滚操作。

### 4.5 桌面记录页

- 使用完整数据表格，补齐记录人和最后修改人。
- 支持日期、状态、金额和文本筛选。
- 记录分页或虚拟化，不一次渲染全部记录。
- 审计历史按需展开或在详情抽屉中分页加载，不在主表下方全部铺开。
- 删除和回滚属于危险操作，必须二次确认并解释影响。

### 4.6 图表分析

图表页采用“总收入趋势 + 每日收入组成”：

- 快捷范围：本月、上月、最近 30 天、自定义。
- KPI：区间总收入、日均收入、营业日数量、最高单日等已有数据可可靠计算的指标。
- 总收入趋势包含历史总额记录和组成模式记录。
- 每日收入组成只展示具有组成明细的记录。
- 只有一个有效组成或没有组成数据时隐藏组成图，不展示无意义饼图。
- 历史总额记录不得被推测或拆分为虚假组成。

### 4.7 管理中心

管理中心划分为：

1. 店铺
2. 用户与权限
3. 收入组成
4. 系统设置
5. 审计记录

每个分区独立加载和分页。管理动作统一使用明确的成功、失败和冲突反馈。

## 5. 收入组成配置

### 5.1 管理方式

管理员以清单方式维护组成项：

- 名称
- 显示顺序
- 是否启用
- 是否计入总收入
- 是否归档

界面实时预览总收入公式，例如：

```text
总收入 = 现金 + 刷卡
外卖平台：仅记录，不计入总收入
```

不提供任意数学公式、减法、比例或脚本表达式，避免金额口径难以理解和审计。

### 5.2 归档与删除

- 组成项可归档，归档后不出现在日常清单和记账页。
- 管理中心提供“已归档”视图，可查看和恢复。
- 从未被任何经营记录引用的误建组成项，可以二次确认后永久删除。
- 已被经营记录引用的组成项不能永久删除，但可以归档。
- 历史记录依赖记录自身保存的快照，不依赖当前组成项是否可见。

### 5.3 配置版本

- 启用、停用、排序、名称或“计入总收入”变化时创建新配置版本。
- 新记录绑定生效版本；旧记录保持原版本快照。
- 历史记录编辑默认使用该记录原有配置快照，避免当前规则重算旧数据。
- 若管理员明确选择“迁移到当前配置”，系统必须说明需要重新填写组成项，并产生审计事件。

### 5.4 建议数据模型

现有具体表名可在实施计划中结合代码确认，逻辑模型如下：

- `income_config_versions`
  - 店铺、版本号、生效时间、创建人、状态
- `income_components`
  - 稳定组成项标识、店铺、当前显示属性、归档状态
- `income_config_version_items`
  - 配置版本与组成项关系、名称快照、顺序、是否计入总收入、是否启用
- `daily_income_items`
  - 经营记录、组成项稳定标识、配置版本、名称快照、计入总收入快照、金额
- `store_daily_records`
  - 保留精确的总收入、收入模式和配置版本引用

金额继续使用数据库定点小数和应用层 `Decimal`，禁止浮点计算。

## 6. 地址与地图交互

用户不再手动填写经纬度。店铺编辑流程为：

1. 输入店名、街道或城市。
2. 显示地址自动完成建议。
3. 选择结果后在地图显示标记。
4. 可使用当前位置。
5. 可拖动标记微调，并进行反向地址解析。
6. 保存人类可读地址以及后台经纬度。

经纬度只作为天气和未来位置相关功能的内部数据。地图或地理编码服务必须封装在独立适配层，供应商、密钥和限额不得散落在页面与业务服务中；实施计划阶段再依据当前服务条款选择具体提供方。

地址不可解析或地图服务不可用时，允许保存地址草稿，但必须明确提示天气定位暂不可用。

## 7. 数据迁移与清理

### 7.1 Sulmona 数据

对现有 Sulmona 记录执行受控迁移：

- 保留日期、收入金额及其他确认无误的数据。
- 全部清除活动状态和活动说明，避免继续展示误填内容。
- 移除“日利润”等利润口径，将记录标记为历史总额收入记录。
- 不为 72 条历史记录虚构收入组成。
- 迁移前生成数据库备份，迁移后核对记录数量、日期范围和总金额。

### 7.2 测试数据

测试夹具店铺不得进入正常生产数据。开发和测试使用独立数据库或显式环境标识；清理当前正常账号可见的 `Slow`、`Failed`、`Healthy` 等测试店铺前，先确认其没有真实业务引用。

### 7.3 回滚历史保留

- 配置回滚最多保留 20 个版本，最长保留 180 天。
- 单条经营记录最多保留 10 个可回滚历史版本，最长保留 365 天。
- 达到版本数量或时间上限后，由清理任务删除最旧的可回滚快照。
- 当前有效记录、当前有效配置和仍被业务记录依赖的必要快照不可清理。
- 精简的审计元数据与大体积回滚快照分离；清理回滚内容不应破坏“谁在何时执行了什么操作”的必要审计链。
- 清理任务必须幂等、可观测，并记录删除数量与失败原因。

## 8. 权限、并发与审计

- 所有权限由后端校验，前端隐藏按钮只用于改善体验。
- 当前管理员不能停用自己的账号。
- 系统不能停用或降权最后一个有效管理员。
- 删除、覆盖、编辑、回滚和配置发布使用版本号或更新时间进行乐观并发控制。
- 客户端提交的记录版本已经过期时返回冲突，不得静默覆盖他人修改。
- 每家店铺每天保持唯一一条经营记录；覆盖必须明确确认并生成前后值审计。
- 记录创建人、最后修改人、删除人、回滚人和配置修改人。
- 组成模式下，审计同时保存输入项、服务器计算结果和使用的配置版本。
- 归档、恢复、永久删除、历史清理和地址变更均属于可审计动作。

## 9. 可扩展架构约束

### 9.1 后端

按领域组织服务与 API：认证、店铺、记账、分析、管理相互独立。人员管理、AI 和自动化以后作为新领域模块接入，不向核心记账服务塞入条件分支。

稳定扩展点包括：

- 店铺范围上下文
- 操作人上下文
- 能力点权限检查
- 经营记录保存后的领域事件
- 后台任务入口
- 分析查询接口

记账事务完成后产生结构稳定的领域事件。当前可以只由审计与简报消费；未来提醒、AI 分析和自动化订阅同一事件，不改变记账 API 的职责。

### 9.2 权限

权限不得只写死为“管理员 / 普通用户”判断。现有角色映射到可命名的能力点，例如查看记录、创建记录、编辑记录、删除记录、查看分析、管理收入组成、管理用户。后续增加店长、工人或只读角色时复用能力点。

### 9.3 前端

路由、侧边栏和“更多”菜单由模块元数据与权限共同生成。页面组件按照领域拆分，共享店铺上下文、权限上下文、错误反馈和响应式布局基础组件。

### 9.4 数据与分析

业务写入模型与分析查询逻辑分开。初期可以共享 MySQL，但接口边界允许未来增加汇总表、缓存或独立分析任务，而不改变记账写入规则。

### 9.5 迁移策略

数据库变更优先采用可兼容的新增、回填、切换、清理步骤。禁止在同一次部署中直接删除旧字段并要求所有代码同时切换。每个迁移需要备份、验证查询和回滚说明。

## 10. 错误、加载与空状态

- 所有异步页面提供骨架或明确加载状态。
- 空状态说明原因并给出下一步操作，而不是只显示空表格。
- 认证失败、权限不足、并发冲突、验证失败、外部服务失败和服务器错误使用不同提示。
- 保存按钮防止重复提交。
- 用户未保存就离开修改页面时提醒。
- 外部天气、地图失败不会导致基础记账不可用。

## 11. 测试与验收

### 11.1 后端

- 总额模式与组成模式计算。
- 不计入总收入项目的保存与计算。
- 配置版本、快照、归档、恢复和删除限制。
- 旧记录不受新配置影响。
- 管理员自停用和最后管理员保护。
- 每店每日唯一性、覆盖确认和并发冲突。
- 回滚版本数量、时间上限与清理任务幂等性。
- Sulmona 数据迁移前后数量、日期范围和金额核对。
- 店铺权限和跨店数据隔离。

### 11.2 前端

- 首页始终是三张卡片和“立即记账”。
- 店铺选择刷新后恢复，失去权限后安全回退。
- 两种收入模式不能同时编辑总额与组成。
- 收入实时计算与后端结果一致。
- 折叠字段保留输入，天气失败仍可保存。
- 手机底部导航不遮挡页面内容与保存按钮。
- 手机记录页无页面级横向滚动，所有操作可触达。
- 桌面记录分页和审计按需加载。
- 图表快捷范围、无组成和单组成状态。
- 地址搜索、选择、拖动和服务失败降级。

### 11.3 真实流程验收

使用管理员账号分别在桌面和手机视口完成：

1. 登录与恢复店铺。
2. 查看首页并进入立即记账。
3. 总额模式记账。
4. 创建收入组成并发布配置。
5. 组成模式记账并验证合计。
6. 覆盖、编辑、删除和回滚。
7. 查看记录人、修改人和审计。
8. 筛选、导出和图表分析。
9. 验证普通用户权限。
10. 验证管理员保护和并发冲突。
11. 验证地图式地址设置与天气定位。
12. 验证本地一键启动脚本。

自动化测试通过不等于验收完成，必须检查真实手机尺寸下的遮挡、滚动、触控区域和错误反馈。

- 人员、工时和工资
- AI 查询、分析、草稿与审批
- 自动提醒、任务编排和长期记忆

这些后续功能不属于本次完成条件，但本设计不得阻塞其接入。

## 13. 实施批次

1. 数据库备份、Sulmona 数据清理和迁移验证框架。
2. 收入组成、配置版本、记录快照、归档及保留策略。
3. 管理员保护、并发控制、能力点权限和店铺选择持久化。
4. 手机导航、首页、紧凑记账页、手机记录详情和桌面侧边栏。
5. 桌面记录表、审计分页、图表、管理中心和地图地址。
6. 响应式、端到端、迁移、权限与回归测试。
7. 管理员桌面与手机全流程验收。
8. 本地启动验证、文档更新和 Git 提交。

每一批都必须先写失败测试或明确验证方式，再实施，并在进入下一批前运行对应验证。

## 14. 非目标

- 不增加 Excel 导入页面。
- 不把系统改造成利润或费用管理系统。
- 不回填历史记录的收入组成。
- 不在本次实现人员管理、AI 或自动化业务。
- 不要求用户输入经纬度。
- 不提供任意收入公式编辑器。
- 不为了扩展性创建无法验证、没有当前消费者的大量空壳服务。

## 15. 完成标准

本设计完成实施需同时满足：

- 本文确认的交互与业务规则全部实现。
- 原一期明确缺口全部补齐。
- Sulmona 活动误填内容已清除且历史金额核对一致。
- 数据迁移可验证、可回滚。
- 权限、并发和保留策略测试通过。
- 桌面与手机真实流程验收通过。
- 后续模块扩展边界有代码结构与测试支撑，而非只存在于文档。
- 工作区无意外文件，相关变更形成可审查的 Git 提交。
