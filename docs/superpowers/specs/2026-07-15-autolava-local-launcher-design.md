# AutoLava AI Windows 本机启动器设计

## 目标

为 Windows 开发机提供一个贯穿 AutoLava AI 全部规划阶段的 PowerShell 一键启动入口。在不安装 Docker 的前提下，复用本机已运行的 MySQL 8.0，自动准备依赖、迁移数据库、初始化管理员、启动 FastAPI 与 Vite，并在系统可用后打开浏览器。

启动器面向本机查看和开发验证，不替代现有 Docker Compose 生产部署方式。

## 用户体验

用户在仓库根目录运行：

```powershell
.\scripts\start-local.ps1
```

首次运行会安装后端和前端依赖，并创建 Git 忽略的本地运行配置。后续运行复用已有环境；只有依赖清单变化时才重新安装。启动器输出中文阶段状态，健康检查成功后打开 `http://127.0.0.1:5173`。

启动器保持在前台监督前后端进程。用户按 `Ctrl+C`、关闭主窗口，或任一子进程异常退出时，启动器终止其创建的其余子进程，不遗留后台开发服务器。

## 架构

### PowerShell 编排器

新增 `scripts/start-local.ps1`，职责限定为：

1. 定位仓库根目录，不依赖调用者当前目录。
2. 检查 Windows、PowerShell、`uv`、Node.js、npm 和 MySQL 服务/端口。
3. 通用读取现有 `.autolava-db.env` 和根目录 `.env` 中的环境变量，但不输出敏感值。
4. 在缺少基础本地配置时补充 JWT、HTTP Cookie 设置和管理员初始化凭据。
5. 准备后端虚拟环境与前端 `node_modules`。
6. 执行 Alembic 迁移和幂等管理员初始化。
7. 启动并监督后端、前端进程。
8. 轮询健康端点，成功后打开浏览器。
9. 在退出路径中清理其创建的子进程。

编排器不安装或修改 MySQL 服务，不修改全局 Python/Node 配置，也不向 Git 写入密钥。

### 本地配置

继续复用已有、被 Git 忽略的 `.autolava-db.env` 作为数据库连接来源。根目录 `.env` 保存以下基础本地值：

- 随机生成且长度不低于 32 字符的 `AUTOLAVA_JWT_SECRET`；
- `AUTOLAVA_COOKIE_SECURE=false`，仅用于回环地址上的本地 HTTP；
- 用户指定的管理员用户名和密码。

脚本只在值缺失时补充配置，不覆盖现有 JWT 或管理员凭据。敏感值不出现在控制台、测试快照、设计文档或受版本控制文件中。

启动器不使用固定的环境变量白名单，而是将两个本地环境文件中的全部 `AUTOLAVA_*` 值传递给后端，且 `.env` 中的显式值可覆盖数据库配置文件中的同名值。Phase 3 引入模型提供商密钥和模型地址、Phase 4 引入调度配置或后续阶段增加其他后端设置时，用户只需更新 `.env`，无需修改启动器。

## 全阶段兼容策略

启动器是仓库级开发入口，不绑定 Phase 1 的页面或数据表：

- Phase 3 的 Agent 与模型客户端仍位于 FastAPI 进程，新增密钥和模型配置通过通用 `.env` 加载；
- Phase 4 的 APScheduler、自动化和记忆引擎仍位于单个 FastAPI 进程，随后端生命周期启动和停止；
- 每次启动都执行 `alembic upgrade head`，因此后续阶段新增迁移不需要改脚本；
- 后端 `pyproject.toml` 和前端 `package-lock.json` 是依赖变化的稳定边界，后续阶段新增包会触发自动更新；
- Vite 的 `/api` 与 `/health` 前缀保持阶段无关，新业务路由无需新增代理规则。

当前路线图没有 Redis、独立任务 Worker 或其他常驻进程，因此不提前构建插件系统。脚本将前置检查、配置加载、依赖准备、迁移、进程启动、健康检查和清理实现为边界清晰的函数；如果未来路线图新增独立服务，可在进程启动/清理清单中加入一个明确组件，而不重写其余流程。

### 后端运行

后端使用 `backend/.venv`。启动器通过 `uv` 创建虚拟环境并安装 `backend/pyproject.toml` 声明的运行依赖。依赖清单发生变化时更新环境。

在 `backend` 工作目录中，启动器向子进程显式传递本地环境变量，依次运行：

1. `alembic upgrade head`；
2. `python -m app.scripts.create_admin`；
3. `uvicorn app.main:app --host 127.0.0.1 --port 8000 --workers 1`。

管理员初始化保持现有幂等语义：账号已存在时不修改密码、角色或启用状态。

### 前端运行与代理

前端通过锁文件安装依赖并运行 Vite 开发服务器：

```text
npm run dev -- --host 127.0.0.1 --port 5173
```

`frontend/vite.config.ts` 增加仅用于 Vite 开发服务器的代理：

- `/api` 转发到 `http://127.0.0.1:8000`；
- `/health` 转发到 `http://127.0.0.1:8000`。

浏览器始终访问 Vite 同源地址，现有相对 `/api` 请求和 Cookie 登录流程无需改动。生产构建与 Nginx 配置不受影响。

## 依赖缓存

启动器在 Git 忽略的本地状态目录中保存依赖清单哈希：

- 后端哈希来源为 `backend/pyproject.toml`；
- 前端哈希来源为 `frontend/package-lock.json`。

虚拟环境或 `node_modules` 不存在、对应哈希变化时安装依赖；否则跳过安装。安装成功后才更新哈希，失败不会把不完整环境标记为可用。

## 启动数据流

```text
前置检查
  -> 读取本地数据库与密钥配置
  -> 按需安装依赖
  -> Alembic 迁移
  -> 幂等创建管理员
  -> 启动 FastAPI
  -> 等待 http://127.0.0.1:8000/health
  -> 启动 Vite
  -> 等待 http://127.0.0.1:5173/health
  -> 打开浏览器
  -> 持续监督两个子进程
  -> 统一清理
```

后端未健康前不启动浏览器。前端代理健康检查通过才视为整个系统就绪。

## 错误处理

启动器遇到以下情况时快速失败，并以中文说明修复动作：

- `uv`、Node.js 或 npm 缺失；
- MySQL80 未运行，或数据库连接失败；
- `.autolava-db.env` 与 `.env` 均未提供数据库 URL；
- 8000 或 5173 端口已被其他进程占用；
- 依赖安装、迁移或管理员初始化失败；
- 后端或前端在就绪前退出；
- 健康检查超时。

错误输出不得包含数据库密码、JWT 或管理员密码。清理逻辑放在 `try/finally` 中，且只终止启动器自己创建的进程。

## 测试与验证

### 自动化测试

新增静态/行为回归，至少验证：

- 启动器存在并使用仓库相对路径；
- 启动器从忽略文件读取密钥，受版本控制文件中不存在真实凭据；
- 启动器通用传递本地 `AUTOLAVA_*` 配置，不把 Phase 3/4 变量硬编码为固定白名单；
- Vite 同时代理 `/api` 与 `/health`；
- 启动顺序包含迁移、管理员初始化、后端健康检查和前端健康检查；
- 子进程清理路径存在且不按进程名全局杀进程；
- `.gitignore` 覆盖本地状态与密钥文件。

### 本机验收

在目标 Windows 电脑上运行启动器，确认：

1. MySQL 复用成功，迁移无错误；
2. 管理员可使用本地配置中的凭据登录；
3. 首页、台账、数据库、图表和管理页面可访问；
4. 刷新页面和 API 请求正常；
5. `Ctrl+C` 后 8000、5173 端口均释放；
6. 再次启动跳过未变化的依赖安装。

同时继续运行现有后端、前端、构建和 Playwright 测试，确保开发代理不会改变生产行为。

## 非目标

- 不安装 Docker Desktop、MySQL、Node.js 或 `uv`；
- 不配置公网访问、HTTPS 或 Windows 防火墙放行；
- 不把本地脚本作为生产服务管理器；
- 不为当前路线图中不存在的 Redis 或独立 Worker 提前实现插件系统；
- 不自动重置已有管理员密码或清空数据库；
- 不改变现有两容器生产发布架构。
