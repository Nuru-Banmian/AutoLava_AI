# AutoLava AI 管理中心优化设计

**日期：** 2026-07-17  
**状态：** 已通过对话确认  
**实现分支：** `feature/phase-1-foundation`

## 1. 背景

现有管理中心包含“收入项目”“用户与权限”“门店设置”“系统状态”四个页签。功能已经具备，但存在以下体验和权限问题：

- 用户卡同时提供卡片外密码输入和编辑区密码输入，入口重复。
- “门店成员”与用户编辑中的门店授权重复。
- 最终管理员与普通管理员使用同一套用户管理操作，缺少所有者保护。
- 门店设置与收入项目分别选择门店，管理上下文割裂。
- 所有用户卡同时铺开，操作密度高，页面层级不清楚。
- 页面展示“普通用户看不到管理中心”的冗余权限提示。

本设计将管理中心调整为以管理对象为中心的工作台，并在服务端建立最终管理员与普通管理员之间的明确权限边界。

## 2. 目标

- 删除重复的密码、门店成员和用户操作历史入口。
- 将门店资料和收入项目整合为同一个门店工作台。
- 使用服务端配置标识并保护最终管理员。
- 允许最终管理员管理其他管理员，禁止普通管理员管理任何管理员。
- 降低页面信息密度，并保持桌面端与手机端一致的操作心智。
- 保留门店资料和收入配置各自独立的保存语义。
- 防止切换管理对象时静默丢失未保存修改。

## 3. 非目标

- 不新增系统状态功能或新的监控数据。
- 不改变数据库管理页已有的历史与回退职责。
- 不在本次处理最终管理员配置缺失、配置错误或对应账号不存在的异常场景。
- 不隐藏最终管理员在数据库记录、审计记录或系统日志中的真实操作者名称。
- 不增加自动保存。

## 4. 方案比较与选择

门店设置与收入项目合并时比较了三种布局：

1. **门店工作台：** 左侧选择门店，右侧同时展示门店资料和收入项目。
2. **合并页内子标签：** 先选择门店，再切换“基本资料”和“收入项目”。
3. **按门店折叠：** 所有门店纵向排列，每家门店展开后显示两类配置。

选择方案 1。它让门店成为唯一上下文，避免重复的门店选择器，并允许管理员同时理解门店资料与收入结构。方案 2 仍然割裂两类配置；方案 3 在门店数量增加后会形成过长页面。

## 5. 页面信息架构

管理中心由四个一级页签缩减为三个，顺序固定为：

1. `门店与收入`，默认页签。
2. `用户与权限`。
3. `系统状态`。

`AdminLayout` 负责页签、页面标题和统一工作区骨架。页面继续使用查询参数保存一级页签；无效参数回退至 `门店与收入`。

视觉语言沿用已确认的门店工作台方案：浅色页面背景、细边框、适度圆角、清晰卡片分区，并区分主要操作、次要操作与危险操作。删除“普通用户看不到管理中心，只能使用已分配门店的日常经营页面。”提示。

## 6. 门店与收入工作台

### 6.1 桌面布局

左侧门店栏包含：

- 门店列表。
- 当前选中状态。
- 门店启用状态。
- “新建门店”入口。

右侧工作区按顺序包含：

1. 门店资料卡。
2. 收入项目卡。

选择门店后，两张卡共同切换到该门店上下文。新建门店使用右侧工作区，不使用弹窗。

门店资料卡在桌面端采用紧凑单行布局，依次展示门店名称、位置摘要、“修改位置”和“保存”；不为常用资料编辑保留过高的纵向空间。窄屏下该行允许自然换行。

### 6.2 保存边界

- 门店资料和收入项目的主按钮文案均为“保存”，两个按钮分别位于各自卡片内。
- 两个区域分别维护草稿、加载、提交、成功和失败状态。
- 任一区域失败不得清空输入，也不得影响另一区域已经成功的保存。
- 收入项目点击“保存”后继续使用现有版本化发布语义，并在发布后失效对应门店的记账、数据库、图表和仪表盘查询；按钮文案简化不改变发布行为。

### 6.3 响应式规则

桌面端显示左侧门店列表。窄屏上，门店列表转换为页面顶部的门店选择器，“新建门店”入口紧邻选择器；门店资料和收入项目在下方纵向排列。

## 7. 用户与权限工作台

### 7.1 页面结构

桌面端左侧显示用户列表和“新建用户”入口。点击用户后，右侧直接显示编辑工作区，不再先显示用户摘要卡。

编辑工作区只包含：

- 角色。
- 账号状态。
- 门店授权。
- 可选的新密码；留空表示不修改。
- 编辑卡片底部操作行左侧的“保存用户”。
- 同一操作行右下角的“永久删除”。

页面不显示独立的危险操作卡片，也不常驻显示“有历史记录只能停用”等删除说明。点击“永久删除”后仍需二次确认；只有后端因历史引用拒绝删除时，才在当前编辑卡片内提示改为停用账号。

删除以下现有内容：

- 用户卡外部的独立密码表单。
- 独立的“门店成员”区域及其前端查询、替换流程。
- 用户操作历史入口及其前端查询流程。历史查看与回退继续由数据库管理页承担。
- 用户摘要小卡片。
- 管理中心可见性提示。

“新建用户”复用右侧工作区，展示用户名、初始密码、角色和门店授权。普通管理员只能创建普通用户；最终管理员可以选择管理员角色。管理员不需要门店授权，提交管理员角色时门店 ID 必须为空。

窄屏上，左侧用户列表转换为顶部用户选择器，编辑表单纵向排列。

### 7.2 未保存修改

用户编辑、新建用户、门店资料或收入项目存在未保存修改时，切换左侧对象必须显示确认：

- “继续编辑”关闭确认并保留当前对象与草稿。
- “放弃修改并切换”清除当前草稿并完成切换。

系统不得自动保存，也不得静默丢弃修改。没有未保存修改时立即切换。

## 8. 最终管理员权限模型

### 8.1 身份来源

服务端使用 `AUTOLAVA_BOOTSTRAP_USERNAME` 识别最终管理员。当前部署值为 `Nuru_Banmian`。前端不得硬编码用户名或直接读取环境变量。

`/auth/me` 增加由服务端计算的 `is_owner: boolean`。前端仅使用该标记决定可见控件；服务端仍是最终授权边界。

### 8.2 列表与展示

- `/admin/users` 不返回最终管理员。
- 最终管理员仅在用户管理列表中隐藏。
- 数据库记录、审计记录和系统日志继续显示其真实用户名。
- 普通管理员能看到其他普通管理员，但只显示只读身份和“管理员”标记。
- 对非最终管理员而言，所有 `role=admin` 的目标账号都只读，包括其自己的管理员账号。

### 8.3 变更规则

最终管理员可以：

- 创建管理员或普通用户。
- 将普通用户提升为管理员。
- 编辑、改密、停用、降级或删除其他管理员。
- 管理普通用户。

普通管理员可以：

- 创建普通用户。
- 编辑、改密、启停或删除普通用户。
- 修改普通用户的门店授权。

普通管理员不可以：

- 创建管理员。
- 将普通用户提升为管理员。
- 编辑、改密、停用、降级或删除任何管理员。

任何管理员均不可以通过用户管理 API 修改、停用、改密或删除最终管理员。现有的“至少保留一个启用管理员”、历史引用保护和不能删除当前账号等约束继续保留，并与新规则共同生效。

## 9. 服务端与前端边界

建议组件边界：

- `AdminLayout`：一级导航与页面骨架。
- `StoreWorkspace`：门店选择、新建和未保存状态协调。
- `StoreDetailsCard`：门店资料编辑。
- `IncomeItemsCard`：收入配置草稿与发布。
- `UsersWorkspace`：用户选择、新建和未保存状态协调。
- `UserEditor`：用户字段、门店授权、密码、底部保存操作和右下角永久删除。
- `UserReadOnlyView`：普通管理员查看管理员身份。
- `SystemStatusPanel`：保留现有数据读取，只统一样式。

权限相关数据流：

1. 登录或刷新会话后，`/auth/me` 返回当前身份及 `is_owner`。
2. 用户工作台读取 `/admin/users`；最终管理员已由服务端过滤。
3. 前端根据 `is_owner` 与目标角色渲染编辑器或只读视图。
4. 创建、修改或删除请求到达服务端后，服务端重新校验最终管理员身份和目标角色。
5. 校验失败返回 `403`；目标状态与业务约束冲突返回 `409`。
6. 成功后只失效受影响的用户和可访问门店查询。

门店成员授权的唯一写入入口为用户创建/修改请求中的 `store_ids`。管理中心删除独立的门店成员读取与整体替换流程。现有服务端 `GET/PUT /admin/stores/{store_id}/members` 端点不在本次删除范围内，以避免扩大对外 API 变更；管理中心及其前端测试不再调用这些端点。

## 10. 错误与状态处理

- 初始加载显示骨架或明确的加载状态。
- 列表加载失败时不显示空列表假象，提供区域级错误。
- 保存失败保留草稿并将错误显示在对应卡片内。
- 权限拒绝使用明确文案，不将其伪装成“用户不存在”。
- 删除因历史引用被拒绝时，才在当前编辑卡片内提示停用账号代替永久删除；正常状态不显示常驻删除说明。
- 提交期间禁用同一区域的冲突操作，但不锁住无关区域。
- 普通管理员不能通过手工请求绕过前端只读状态；服务端返回 `403` 且不产生审计变更。

## 11. 测试与验收

### 11.1 服务端

- `/auth/me` 对最终管理员返回 `is_owner=true`，其他账号返回 `false`。
- `/admin/users` 对所有管理员均排除最终管理员。
- 针对最终管理员的修改和删除请求全部拒绝。
- 普通管理员不能创建管理员或提升普通用户。
- 普通管理员不能修改、改密、停用、降级或删除任何管理员。
- 最终管理员可以管理其他管理员。
- 普通管理员仍能完整管理普通用户及其 `store_ids`。
- 被拒绝的请求不修改数据库，也不生成成功审计。

### 11.2 前端

- 一级页签顺序为“门店与收入、用户与权限、系统状态”，默认进入“门店与收入”。
- 用户密码输入只在创建或编辑工作区出现一次。
- 页面不存在独立门店成员栏、用户操作历史、用户摘要卡和管理中心可见性提示。
- 最终管理员看不到自己的用户项，但能编辑其他管理员。
- 普通管理员看到管理员时只有只读视图。
- 普通管理员的新建表单没有管理员角色选项。
- 门店资料在桌面端为紧凑单行，名称、位置摘要、修改位置和保存操作位于同一行；窄屏允许自然换行。
- 门店资料卡和收入项目卡的主按钮文案均为“保存”，收入保存仍执行版本化发布。
- 门店资料和收入配置独立保存及独立报错。
- 用户编辑页没有独立危险操作卡片或常驻删除说明；“永久删除”位于编辑卡片右下角，并保留二次确认与删除失败后的停用建议。
- 有未保存修改时，切换用户或门店必须确认。
- 放弃修改会切换对象；继续编辑会保留原对象和草稿。
- 桌面端使用左右工作台，手机端使用顶部选择器和纵向内容。

### 11.3 回归

- 用户创建、编辑、停用和安全删除规则继续通过。
- 门店创建、地图定位、编辑、停用和安全删除继续通过。
- 收入项目排序、归档、恢复、永久删除和版本发布继续通过。
- 系统状态数据、时间戳和告警展示逻辑保持不变。
- 数据库管理页的历史查看和回退能力保持不变。

## 12. 完成标准

当页面结构、权限规则、未保存保护、响应式行为和上述自动化测试全部通过时，本次管理中心优化完成。实现不得引入第二套门店授权入口，也不得将最终管理员保护仅实现为前端隐藏。
