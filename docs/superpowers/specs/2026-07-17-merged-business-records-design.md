# AutoLava AI 营业记录合并页面设计

**日期：** 2026-07-17  
**状态：** 已通过对话与交互原型确认  
**设计分支：** `codex/merge-analysis-history`  
**基线：** `feature/phase-1-foundation`

## 1. 背景

一期当前将“历史记录”和“经营分析”拆成两个页面。历史记录使用占据大量空间的月历，选中日期后的记录详情反而较小；经营分析需要通过另一个导航入口打开，用户无法在浏览每日记录时同时理解经营趋势。

本设计将两页合并为一个“营业记录”页面。页面以可分页的 Excel 式记录表为主要内容，同时在右侧展示选中记录详情和经营分析。已通过浏览器交互原型确认桌面宽度比例、记录选择、右栏布局、统一分析卡片和手机端详情面板。

## 2. 目标

- 用按日期排列的表格替代历史月历，提高每日记录的浏览密度。
- 在同一页面同时提供单日详情、营业额趋势和收入构成。
- 保持记录筛选与分析筛选互不影响，避免查看动作相互打断。
- 保留修改、补记、导出、删除、审计和回滚能力及其原有权限边界。
- 保留收入构成，并明确区分计入与不计入总营业额的项目。
- 在桌面和 320px 手机宽度上都避免页面级横向滚动和不可触达操作。
- 删除重复的前端经营分析页面和导航入口。

## 3. 非目标

- 本分支不重命名任何后端 API 路径。
- 本分支不删除 `/api/database/...` 或 `/api/charts/...` 数据接口。
- 本分支不重新设计收入项目管理；管理中心优化继续由 `codex/management-center-optimization` 完成。
- 本分支不改变收入配置的版本化发布、归档、恢复或删除规则。
- 本分支不增加收入预测、AI 解释或新的经营指标。
- 本分支不实现任意列排序、列拖动、列显隐配置或通用电子表格编辑。

API 路径整理应在管理中心优化与本页面合并都完成后，另开独立分支统一处理。

## 4. 页面身份与导航

合并页面名称固定为“营业记录”，继续使用前端路径 `/database`。

导航调整：

- 桌面侧栏删除“历史记录”和“经营分析”，在原位置只保留“营业记录”。
- 手机底部“记录”继续指向 `/database`。
- 手机“更多”删除“经营分析”。
- 前端 `/charts` 路由直接删除，不提供重定向。
- `ChartsPage` 不再作为页面存在；可复用的图表组件保留并由合并页面调用。

删除 `/charts` 只影响前端页面路由。内部 `/api/charts/{store_id}` 数据接口仍为经营分析提供数据。

## 5. 桌面页面结构

### 5.1 总体布局

桌面使用两栏布局：

- 左栏为主要区域，展示历史筛选、记录表格、导出和分页。
- 右栏约占页面内容区的三分之一，上方为选中记录详情，下方为一张统一经营分析卡。
- 桌面断点使用 `minmax(0, 1fr) minmax(22rem, 24rem)` 两列；实现沿用现有页面最大宽度和设计令牌，不硬编码业务色值。

右栏作为一个整体吸附在可视区域顶部。若详情和分析合计高度超过当前视口，右栏允许自身纵向滚动；左侧表格仍随页面正常滚动，不创建第二个表格纵向滚动容器。该行为防止用户点击靠下记录后还要回到页面顶部查看详情。

### 5.2 历史筛选与分页

左栏默认状态：

- 时间范围为当前门店本月。
- 日期按最新到最旧排列。
- 每页固定 15 条，不提供页大小选择器。
- 筛选提供“本月”“上月”“自定义”。
- 导出使用当前左栏筛选范围，不使用右侧分析范围。

切换范围后回到第一页，并自动选中当前页第一条记录。翻页后同样自动选择新页第一条记录。

### 5.3 桌面表格

左栏表格只显示稳定、便于快速扫描的列：

1. 日期。
2. 营业状态。
3. 总营业额。
4. 天气。

不在表格中展示动态收入分类、洗车数量或活动摘要。收入分类会随门店配置新增、改名、排序和归档，不适合作为长期稳定的表格列；洗车数量和活动摘要属于单日详情。

选中行使用主色背景或边标记，同时保留文字和状态圆点，不能只靠颜色表达。点击任意行只更新右上详情，不改变左侧范围、页码或右下分析范围。

### 5.4 记录详情

首次进入页面、切换门店、切换范围或翻页后，自动展示当前页最新记录。详情包含该记录实际存在的数据：

- 日期与营业状态。
- 总营业额。
- 各收入项目及其历史名称、金额和计入规则。
- 洗车数量。
- 天气。
- 活动摘要。
- “修改这天记录”。

管理员额外看到删除、审计历史和回滚入口；普通用户不看到这些管理操作。休息或天气停业记录必须显示真实状态，不能沿用上一条记录的“营业”标签。

## 6. 统一经营分析卡

右下只有一张“经营分析”卡。它由一套时间选择共同驱动以下全部内容：

1. 区间指标。
2. 营业额趋势曲线。
3. 营业天数和营业日均。
4. 收入构成。

收入构成不是独立卡片，也没有独立时间选择。任何范围变化都必须以相同起止日期一次性刷新整张卡片。

### 6.1 分析范围与比较口径

分析范围与左侧记录范围互不影响，默认同样为本月。规则固定为：

- **本月：** 本月至今对比上月同期，按日绘制趋势。
- **上月：** 完整上月对比再前一个完整月，按日绘制趋势。
- **近 6 月：** 主区间从当前月往前第五个月的 1 日开始，到门店本地今天结束；比较区间整体向前平移六个月，结束日保持相同日号，遇到短月时取该月最后一天。两段区间按月绘制趋势。
- **自定义：** 不自动生成对比，只展示所选范围的总额、营业天数、营业日均、趋势和收入构成。
- 自定义范围不超过 62 天时按日绘制；超过 62 天时按月汇总。

“本月至今”不得直接与完整上月比较。比较区间的起止日期必须显示或通过可访问文本说明，避免用户误解百分比口径。

### 6.2 收入构成

收入构成是必须保留的能力，分类名称和顺序不能硬编码。它们来自当前门店的版本化收入配置与历史记录项目。

同一区域分为两组，中间使用清晰横线分隔：

#### 计入总营业额

- 展示每项金额和在“已分类且计入营业额的金额”中的占比。
- 百分比分母是所选范围内有分类明细的计入项合计，不是包含旧总额记录的区间总营业额。
- 当可比较的计入分类少于两个时，不绘制无意义的构成比例图；保留清楚的单项或空状态说明。

#### 不计入总营业额

- 展示每项记录金额。
- 不显示占总营业额百分比。
- 明确说明这些项目独立记录，不参与区间总营业额、增减金额或营业日均计算。

历史“仅总额”记录继续参与总营业额、趋势和营业日均，但不得被推测或分配到任何收入分类。因此“已分类营业额”可以小于区间总营业额，界面必须明确说明这一点。

每组默认按配置顺序展示前 5 项；超过 5 项时在该组内部提供“查看全部/收起”。不得静默合并为一个无法追溯的“其他”分类。

## 7. 手机布局

手机不压缩桌面四列表格，也不产生页面级或表格级横向滚动。

### 7.1 记录列表

手机使用三列紧凑列表：

1. 日期。
2. 营业状态。
3. 总营业额。

天气移入单日详情。筛选、每页 15 条和日期倒序规则与桌面一致。

### 7.2 单日详情

点击记录后从页面底部打开详情面板：

- 面板显示与桌面相同的完整记录内容和权限操作。
- 面板预留手机底部导航安全区。
- 关闭面板后保持原页码、筛选、滚动位置和选中行。
- 打开详情时允许暂时遮住列表，这是为了避免用户点击靠下记录后再滚回详情位置。

### 7.3 经营分析

经营分析位于记录列表和分页下方，使用与桌面相同的一张统一卡片。指标、图表、图例、分类名称和金额纵向重排，不发生横向滚动。

## 8. 前端组件边界

组件边界固定如下：

- `BusinessRecordsPage`：门店上下文、桌面/手机布局、两套独立筛选与查询协调。
- `RecordFilters`：记录范围、自定义日期与导出。
- `RecordTable`：桌面四列表格、分页和选中状态。
- `MobileRecordList`：手机三列列表。
- `RecordDetailPanel`：桌面详情内容和权限操作。
- `MobileRecordSheet`：复用详情内容的手机容器。
- `BusinessAnalysisCard`：分析范围、比较指标、趋势和统一加载/错误边界。
- `IncomeComposition`：计入与不计入两组展示。
- `ChartPanel`：继续作为可复用图表渲染单元，不拥有页面级查询状态。

组件必须以明确 props 传递数据和事件。表格、详情和分析不各自读取全局门店状态，避免切换门店时出现不同步数据。

## 9. 数据流与接口边界

### 9.1 本分支保留的路径

- 记录列表、导出、审计和回滚继续使用 `/api/database/...`。
- 分析继续使用 `/api/charts/{store_id}`。
- 记账修改继续使用现有 `/api/ledger/...`。

路径重命名不属于本分支。前端查询键也继续使用当前 `database` 和 `charts` 语义，以避免与正在实现的管理中心收入发布失效逻辑发生交叉。

### 9.2 记录查询

记录接口按左栏范围请求 `start`、`end`、`page` 和固定 `page_size=15`，并返回最新日期优先的分页结果。若现有服务默认顺序不同，本分支需同步调整服务与测试。

### 9.3 分析查询

分析仍通过 `/api/charts/{store_id}` 提供。合并页在现有 `start`、`end` 基础上增加可选的 `compare_start`、`compare_end` 和必填的 `bucket=day|month`。`compare_start` 与 `compare_end` 必须同时出现，否则返回 422；自定义范围不发送这两个参数。这属于既有接口的加法扩展，不是路径整理。

现有 `ChartsResponse` 的 `kpis`、`daily`、`monthly` 和 `categories` 字段继续保留；新增字段为：

- `range`：当前区间起止日期与 `day|month` 聚合粒度。
- `comparison_kpis`：比较区间起止日期、总营业额、营业天数和营业日均；自定义范围为 `null`。
- `excluded_categories`：不计入总营业额的分类金额及配置顺序。
- `classified_included_total`：计入项已分类合计，用于构成占比的准确分母。

现有 `categories` 继续表示计入总营业额的分类，现有 `daily` 与 `monthly` 继续提供两种序列；前端根据 `bucket` 只渲染其中一种。新增字段保持管理中心与其他当前调用方兼容。

服务端从真实记录项目聚合分类，不允许用总营业额减法或比例推测分类金额。被历史记录引用的已归档分类仍保留在相关范围的构成结果中。

### 9.4 查询失效

修改、补记、删除或回滚记录后，失效当前门店的记录、详情、分析和首页查询。收入配置发布后继续失效记账、记录、分析和首页查询。

管理中心优化分支完成后，本分支在进入产品代码实现前必须同步最新基线，并验证其收入发布失效逻辑仍覆盖 `database` 与 `charts` 查询键。

## 10. 加载、空状态和错误

- 左侧记录和右侧分析分别请求、缓存、重试和报错；任何一侧失败不得遮住另一侧成功内容。
- 切换门店后，记录恢复本月第一页，分析恢复本月，清除旧门店选中记录。
- 记录加载显示表格骨架，不显示上一门店数据。
- 当前记录范围为空时显示明确空状态和补记入口；右上显示“暂无可查看记录”。
- 分析范围为空时保留范围标题，指标显示真实零值，图表显示“该范围暂无经营数据”，不生成假曲线。
- 分析请求失败时在统一经营分析卡内显示错误和重试，不影响记录表和详情；若已有同门店、同范围的成功缓存，则保留旧数据并明确标记刷新失败。
- 导出失败提供中文反馈，不改变当前筛选。
- 版本冲突、权限不足和普通服务错误继续使用现有不同中文提示。
- 切换门店、筛选或页码时，迟到响应不得覆盖当前门店或当前范围。

## 11. 可访问性与视觉规则

- 表格使用真实表格语义；选中行同时提供 `aria-selected` 和可见样式。
- 行必须可通过键盘选择，不能只监听鼠标点击。
- 营业状态同时使用文字和状态标记。
- 趋势图提供区间、粒度和关键数值的文本等价内容。
- 收入构成条形不能只靠颜色区分类别，始终显示分类名称和金额。
- 手机底部详情使用可访问对话框或底部面板语义，打开后管理焦点，关闭后将焦点还给原记录行。
- 加载使用 `role="status"`，错误使用 `role="alert"`，重试按钮具有明确名称。
- 所有颜色、圆角、阴影和间距使用现有主题令牌。

## 12. 测试与验收

### 12.1 后端

- 记录列表按日期倒序并固定支持 15 条分页。
- 门店隔离和角色权限不变。
- 本月、上月、近 6 月和自定义范围生成正确起止日期与日/月聚合。
- 本月至今只与上月同期比较。
- 自定义范围不返回自动比较。
- 计入与不计入分类正确分组、排序和合计。
- 旧总额记录影响总额和趋势，但不影响任何构成金额。
- 已归档且被历史记录引用的分类仍能出现在相关历史范围。
- 删除、审计、回滚和导出回归通过。

### 12.2 前端单元与集成

- 页面名称和桌面导航只剩“营业记录”。
- 手机“更多”不再出现“经营分析”。
- `/charts` 前端路由和 `ChartsPage` 被删除。
- 默认本月、日期倒序、每页 15 条和自动选中最新记录。
- 左表筛选与分析筛选互不影响。
- 点击营业、休息和天气停业记录时详情状态同步正确。
- 切换分析范围时 KPI、趋势和两组收入构成使用同一起止日期刷新。
- 不计入项没有百分比，也不进入总营业额指标。
- 两侧独立加载、错误和重试。
- 管理员与普通用户操作可见性正确。

### 12.3 浏览器验收

- 桌面滚动左侧记录时右栏保持可访问，选中行立即更新详情。
- 右栏内容超过视口时可独立滚动到完整收入构成。
- 手机点击记录打开底部详情，关闭后恢复原滚动位置。
- 320px 宽度没有页面级或表格级横向滚动。
- 手机底部导航不遮挡详情操作或页面末尾内容。
- 切换月份、自定义范围和翻页不会出现旧范围数据闪回。
- 完成从记录查看、修改、导出到经营分析的整页流程。

## 13. 集成顺序

为了避免与正在实现的管理中心优化产生不必要冲突：

1. 完成并合并 `codex/management-center-optimization`。
2. 当前分支同步该最新基线。
3. 再实施本设计的产品代码。
4. 验证管理中心发布收入配置后仍刷新记录与分析。
5. 两个功能分支完成后，另开 API 路径整理分支。

本设计文档可以先提交到当前分支；同步基线时保留该文档并根据实际合并结果更新实施计划，不提前修改管理中心 worktree。

## 14. 完成标准

- 用户只通过“营业记录”进入历史与分析能力。
- 桌面表格、详情和统一分析卡与确认原型一致。
- 手机紧凑列表、底部详情和纵向分析通过真实视口验收。
- 所有分析范围和比较口径准确、可解释。
- 收入构成完整保留并按计入规则分组，历史总额不被虚构拆分。
- 旧前端 `/charts` 页面完全删除，既有后端 API 路径保持不变。
- 记录、导出、修改、删除、审计和回滚能力无回归。
- 相关后端、前端和浏览器测试全部通过。
