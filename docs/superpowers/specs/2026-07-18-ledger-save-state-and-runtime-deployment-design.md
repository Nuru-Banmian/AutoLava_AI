# 每日记账保存状态与运行版本部署设计

> 状态：待用户复核
> 日期：2026-07-18

## 背景与根因

服务器目录中的源码已经更新到包含月度记录标记的版本，但运行中的 Web 镜像创建时间早于该功能提交时间。上次镜像构建因服务器失去响应而未完成，重启后 Docker 仅重新拉起了旧镜像。因此线上旧前端不会请求当前浏览月份的完整记录列表，只能用当前选中记录或最近记录生成有限标记。

线上复现同时确认：`sulmona` 门店的 2026 年 4 月 26、27、28、30 日单日记录均可读取；其中 4 月 28 日自动回填营业额 `613.00`，但旧前端打开 4 月日历时未发出月度列表请求，仅将当前选中的 4 月 30 日标记为已有记录。

保存体验问题来自未保存状态的清理契约不够明确。当前实现依赖表单的异步签名比对间接清除全局脏状态。在保存完成、记录重新获取和用户切换日期交错时，页面可能在已经提示“保存成功”后仍保留未保存标志。

## 目标

1. 保存成功后，如果用户在保存期间没有继续修改表单，切换日期、门店或页面时不再显示“放弃未保存的修改”提示。
2. 如果用户在保存请求进行期间又修改了字段，新修改仍视为未保存，继续受到离开提醒保护。
3. 日历打开时请求当前浏览月份的完整记录列表；所有非选中且已有记录的日期在日期下方显示蓝点。
4. 选中日期继续使用蓝色选中背景区分；如果该日期已有记录，辅助文本和无障碍名称继续明确显示“已有记录”，不要求在蓝色选中背景上再显示蓝点。
5. 部署必须证明运行中的镜像来自目标提交，不能只证明服务器目录源码已经更新。

## 保存状态设计

表单为每次提交生成一个规范化数据签名，并记录提交时的编辑版本。保存成功后按以下规则处理：

- 当前表单签名仍等于提交签名：立即把提交内容设为新的已保存基线，并明确清除全局未保存状态。
- 当前表单签名已不同于提交签名：说明用户在保存期间继续编辑；保留当前输入和未保存状态，不覆盖新编辑。
- 后续服务端规范记录返回时，仅当当前表单仍等于已保存基线时吸收服务端数据；不得覆盖保存期间产生的新编辑。
- 保存失败或发生覆盖冲突时，不更新已保存基线，继续保留未保存状态。

日期选择仍通过统一的未保存变更守卫执行。守卫不增加特殊绕过逻辑，而是消费表单提供的准确脏状态。

## 月度记录标记设计

沿用现有按 `门店 ID + 月份` 隔离的查询缓存：

1. 日期选择器打开时报告当前显示月份。
2. 页面请求该月第一天至最后一天的营业记录，单页容量覆盖一个自然月。
3. 返回记录日期转换为集合并传入日历。
4. 非选中且存在于集合中的日期显示清晰可见的蓝点。
5. 保存成功后失效对应门店的月份查询；下次打开日历时重新获取标记。
6. 月度查询失败时不推测标记，但单日选择、读取和修改仍可继续使用。

## 部署与运行版本校验

发布继续使用 `test-used` 分支并更新现有 PR。服务器更新遵循以下顺序：

1. 在本地完成针对性测试、前端全量测试、生产构建和后端全量测试。
2. 推送 `test-used`，更新 PR。
3. 部署前创建数据库备份并确认数据卷存在。
4. 上传目标提交源码，在服务器执行无缓存或具备明确镜像重建证据的 Web/API 构建。
5. 记录目标 Git 提交，并核对运行容器的镜像创建时间与静态资源内容。
6. 通过 HTTPS 健康检查，并在真实线上页面验证 `sulmona` 的 2026 年 4 月 26、27、28、30 日均被识别为已有记录。
7. 验证修改并保存一条记录后，未再次编辑时可直接切换日期；保存期间继续编辑时仍出现保护提示。

部署不得删除或覆盖服务器 `.env`、`backups/` 和 MySQL 数据卷。直接公网 8080 不是验收入口，生产验收使用 `https://nuru-banmian.cn`。

### 低内存服务器构建路径

服务器实测无法稳定完成包含 Node.js、`npm ci` 和 Vite 的多阶段 Web 构建，即使 Web 与 API 已严格串行，单独的 Web 构建仍会使 SSH 失去响应。因此生产发布增加受测的低内存路径：

1. 前端测试和 `npm run build` 在本地完成。
2. 本地 `dist/` 与目标提交源码分别上传；服务器不执行 Node.js 构建。
3. 使用独立的预构建静态文件 Dockerfile，只基于缓存的 Nginx 运行时镜像复制 `nginx.conf` 和 `dist/`，同时写入同一个 OCI Git revision 标签。
4. 本次后端运行代码没有变化，因此不重建 API 镜像；只强制重建并替换 Web 容器。
5. 服务器任意时刻最多运行一个镜像构建任务。
6. 新 Web 容器仍必须通过 OCI 标签、`version.json`、HTTPS 健康检查和真实日历只读验收。

该路径不降低发布门槛：本地生产构建必须先成功，上传后的 `dist/` 必须来自已验证工作树，服务器运行版本必须与目标 Git SHA 完全一致。

## 测试与验收

- 单元/组件测试：现有记录保存成功后，立即选择另一日期，不出现未保存提示。
- 单元/组件测试：保存请求未完成时继续编辑；请求成功后选择另一日期，仍出现未保存提示且新输入保留。
- 单元/组件测试：月度接口返回多个历史日期时，所有非选中记录日期均具有 `data-recorded` 标记和蓝点。
- 集成测试：打开历史月份会发出正确的月份起止日期请求，并在保存后刷新对应月份缓存。
- 线上验收：4 月 26、27、28、30 日全部显示为已有记录；选择其中任意日期均自动回填已保存数据。
- 运行版本验收：容器内静态资源包含月度请求逻辑，镜像创建时间晚于目标提交或附带等价的目标提交标识。

## 非目标

- 不修改营业记录数据或迁移历史数据。
- 不改变日期选择器的整体布局、日期禁用规则或门店权限。
- 不对无记录日期生成推测性标记。
- 不在本次修复中重构其他页面的全局导航守卫。
