# AutoLava AI — 户外洗车房智能管理系统 · 设计文档

> 版本：v2.0 | 日期：2026-07-13

---

## 一、项目概述

**AutoLava AI** 是一套面向意大利户外洗车房场景的家庭经营管理系统。系统服务于小型家庭协作经营：家人作为普通用户负责日常记账、工时填写、查看数据；店主作为管理员负责用户、店铺、权限、收入项目、系统提醒和复杂维护。

系统第一目标不是做复杂企业后台，而是让家人可以用尽量简单的表单完成真实经营记录。AI 是增强能力，用于查询、分析、草稿生成和审批式修改，但不作为基础记账的唯一入口。

### 核心原则

- 人工记账是主流程，AI 是增强流程。
- 普通用户界面要简单，管理员界面可以承载复杂配置。
- 店铺、用户、收入项目、工人工时都按权限隔离。
- 所有关键数据变更可审计、可回退。
- 天气和活动用于辅助分析，不影响基础记账可用性。
- 第一阶段先把非 AI 的经营系统做稳，再分阶段加入工时工资、AI 和记忆。

## 二、技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 后端框架 | FastAPI | Python 异步框架，提供 REST API 和 OpenAPI 文档 |
| ORM / Migration | SQLAlchemy 2.x + Alembic | 管理 MySQL 表结构和版本迁移 |
| 前端 | React + shadcn/ui + Tailwind CSS | 响应式界面，适配桌面端和手机端 |
| 图表 | Recharts | React 生态轻量图表库 |
| 数据库 | MySQL | 安装在服务器系统中，不作为 Docker 容器运行 |
| Agent 框架 | LangChain + LangGraph | 用于 AI 工具调用、结构化计划和审批流程 |
| LLM 主模型 | DeepSeek-V3 / DeepSeek-R1 | DeepSeek-V3 负责常规对话，DeepSeek-R1 负责复杂推理 |
| LLM 备用 | Qwen-Plus | DeepSeek 不可用时备用 |
| 天气服务 | Open-Meteo | 免费、无需 API Key，支持预报、历史天气、地理编码 |
| 天气备用 | 和风天气 | 预留备用接口 |
| 定时任务 | APScheduler | 第一版内置在 FastAPI 单进程中，后续可拆 worker |
| 部署 | 阿里云 + Docker | Docker 部署后端和前端，MySQL 在服务器系统中独立运行 |
| 版本管理 | GitHub | 分支管理和备份，不直接参与服务器部署链路 |

## 四、角色、权限与店铺访问

### 4.1 角色

| 角色 | 使用者 | 说明 |
|---|---|---|
| 管理员 | 店主 | 管理所有用户、店铺、授权、收入项目、系统提醒、任务日志和全部数据 |
| 普通用户 | 家人 | 管理自己有权限的店铺数据，进行记账、查看、修改、删除、回退、导出 |

### 4.2 权限规则

- 普通用户不开放注册，由管理员创建。
- 用户和店铺通过 `store_members` 关联。
- 第一版用户-店铺权限只做“有权限 / 无权限”，不细分只读和编辑。
- 有权限的普通用户可以查看、记账、修改、删除、回退、导出该店铺数据。
- 无权限店铺对普通用户完全不可见。
- 管理员可以查看和管理所有店铺。
- AI 只能操作当前选中的店铺，不能跨店铺修改数据。
- 管理员可跨店铺分析；普通用户只能分析自己有权限的店铺。

### 4.3 启用与停用

- 用户不做物理删除，采用启用 / 停用。
- 店铺不做物理删除，采用启用 / 停用。
- 收入项目已有历史数据时，不允许硬删除，只允许停用。
- 工人不做硬删除，只允许停用。

## 五、数据库设计

### 5.1 users — 用户表

| 字段 | 说明 |
|---|---|
| id | 主键 |
| username | 登录账号，唯一 |
| password_hash | bcrypt 密码哈希 |
| role | `admin` / `user` |
| is_active | 是否启用 |
| remember_token | 记住我 token，可按实现调整 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

### 5.2 stores — 店铺表

| 字段 | 说明 |
|---|---|
| id | 主键 |
| name | 店铺显示名，用户可见，管理员可修改 |
| address | 地址文本 |
| latitude | 纬度 |
| longitude | 经度 |
| timezone | 时区，如 `Europe/Rome` |
| is_active | 是否启用 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

说明：

- 两个实际店铺均位于意大利，但仍按店铺存储独立时区。
- 店铺地址、经纬度和时区由管理员维护。
- 店铺改名不影响历史数据。
- 店铺地址改变后，不自动修改历史天气记录，只影响后续查询。

### 5.3 store_members — 用户店铺关联表

| 字段 | 说明 |
|---|---|
| id | 主键 |
| store_id | 店铺 ID |
| user_id | 用户 ID |
| created_at | 创建时间 |

唯一约束：

```text
store_id + user_id 唯一
```

### 5.4 store_settings — 店铺设置表

| 字段 | 说明 |
|---|---|
| store_id | 店铺 ID，主键或唯一 |
| standard_work_hours | 一天标准工作时长，整数，范围 1-24 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

`standard_work_hours` 用于工时折算和工资结算。

### 5.5 income_categories — 收入项目配置表

| 字段 | 说明 |
|---|---|
| id | 主键 |
| store_id | 所属店铺 |
| name | 收入项目名，如现金、刷卡、明钱、暗钱 |
| include_in_total | 是否计入总收入 |
| is_active | 是否启用 |
| sort_order | 排序 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

规则：

- 每个店铺可以配置不同收入项目。
- 普通用户只填写金额，收入项目配置由管理员维护。
- 总收入只由 `include_in_total = true` 的收入项目自动计算。
- 普通用户不能手动修改总收入。
- 停用后的收入项目不再出现在新记账表单，但历史数据仍可查询。

### 5.6 store_daily_records — 每日经营主记录表

统一使用一张每日记录表，不再使用 `store_data_{店铺名}` 动态表。

| 字段 | 说明 |

| weather | 最终天气，用户可修改，用于展示和分析 |
| weather_auto | API 自动天气，后台保留 |
| weather_code | 天气代码 |
| temperature_max | 最高温 |
| temperature_min | 最低温 |
| precipitation | 降雨量 |
| activity | 活动备注 |
| scanned | 是否已被记忆引擎扫描 |
| created_by | 创建用户 |
| updated_by | 最后修改用户 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

唯一约束：

```text
store_id + date 唯一
```

### 5.7 daily_income_items — 每日收入明细表

| 字段 | 说明 |
|---|---|
| id | 主键 |
| record_id | 每日主记录 ID |
| category_id | 收入项目 ID |
| amount | 金额 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

总收入计算：

```text
daily_revenue = sum(amount where income_categories.include_in_total = true)
```

每次收入明细新增、修改、删除后，后端重新计算并保存 `daily_revenue`。

### 5.8 workers — 工人表

| 字段 | 说明 |
|---|---|
| id | 主键 |
| store_id | 所属店铺 |
| name | 工人姓名 |
| daily_full_wage | 做满一个标准工作日的工资 |
| is_active | 是否启用 |
| sort_order | 排序 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

### 5.9 worker_time_records — 工人工时记录表

| 字段 | 说明 |
|---|---|
| id | 主键 |
| store_id | 店铺 ID |
| worker_id | 工人 ID |
| work_date | 工作日期 |
| hours | 当天工时，`null` 表示未填写，0 表示明确 0 小时 |
| created_by | 创建用户 |
| updated_by | 最后修改用户 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

唯一约束：

```text
worker_id + work_date 唯一
```

工时规则：

- 输入方式为滚动选择。
- 可选 `未填写`、`0 小时`、`1 小时` ... `24 小时`。
- 第一版只支持整数小时，不支持小数。
- 不记录上下班时间。
- 不记录备注。

### 5.10 worker_payrolls — 工资结算主表

| 字段 | 说明 |
|---|---|
| id | 主键 |
| store_id | 店铺 ID |
| month | 结算月份 |
| standard_work_hours | 结算时的标准工作时长快照 |
| total_amount | 本月工资总额 |
| status | `active` / `superseded` |
| created_by | 生成人 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

### 5.11 worker_payroll_items — 工资结算明细表

| 字段 | 说明 |
|---|---|
| id | 主键 |
| payroll_id | 工资结算 ID |
| worker_id | 工人 ID |
| worker_name_snapshot | 工人姓名快照 |
| total_hours | 月总小时 |
| work_days | 上班天数 |
| standard_days | 折合标准工作日 |
| daily_full_wage_snapshot | 满标准日工资快照 |
| amount | 工资金额 |
| created_at | 创建时间 |

工资计算规则：

```text
当天工资 = min(当天小时数, 标准工作时长) / 标准工作时长 * 满标准日工资
月工资 = 当月每天工资相加
```

超过标准工作时长的部分不增加当天工资。

结算生成后保存快照。结算后如果历史工时被修改，系统提示当前工时与已结算工资不一致，并允许重新生成结算。旧结算保留并标记为 `superseded`。

## 六、页面设计

### 6.1 登录页 `/login`

- 未登录访问任何页面时跳转到登录页。
- 不提供普通用户注册入口。
- 支持“记住我”。
- 登录成功后进入首页。

### 6.2 首页 `/`

首页保持轻量，面向家人日常查看。

核心内容：

- 店铺选择器。
- 昨日总结。
- 今日播报。
- 明日预告。
- 明显的“去记账”按钮。
- 手动刷新 AI 摘要按钮，需做频率限制。

三卡片规则：

| 卡片 | 内容 |
|---|---|
| 昨日总结 | 昨日是否记账、总收入、主要收入项目、天气、洗车数量、活动备注 |
| 今日播报 | 今日天气、今日是否已记账、记忆规律提示 |
| 明日预告 | 明日天气、周几、记忆规律提示 |

规则：

- 每天凌晨 4 点自动生成。
- 用户打开首页直接读取，不等待 AI。
- 昨日数据补录或修改后，单独重新生成昨日总结。
- 今日已记账时，今日卡片显示今日收入。
- 第一版不放最近 7 天趋势图。
- 文案简短、温和、不做绝对预测、不使用具体百分比预测。

### 6.3 记账页 `/ledger`

记账页是家人最高频使用页面，主流程是人工表单。

字段：

- 店铺：用户只有一个店铺时自动选中。
- 日期：默认今天，不能选未来日期。
- 营业状态：`营业` / `休息` / `天气停业`。
- 收入项目：按当前店铺配置动态显示。
- 总收入：系统自动计算，不允许手动改。
- 天气：自动填入，可手动修改。
- 洗车数量：可选。
- 活动备注：可选。

收入项目示例：

```text
现金 200 欧元，计入总收入
刷卡 150 欧元，计入总收入
明钱 100 欧元，仅记录
暗钱 80 欧元，仅记录
总收入 350 欧元，自动计算
```

营业状态规则：

- `休息`：收入项目默认 0，洗车数量默认 0，可锁定输入。
- `天气停业`：默认 0，但不锁死，允许手动改。
- `营业`：正常填写。

重复日期规则：

- 同一店铺同一天已有记录时，提示是否覆盖。
- 覆盖保存写入审计日志。

辅助区：

- 显示最近 7 天记录，方便确认是否漏填或填错。
- 可从最近记录进入修改。

### 6.4 数据库管理页 `/database`

普通用户可进入，只能管理有权限店铺数据。管理员可查看所有店铺。

功能：

- 店铺选择。
- 日期范围筛选。
- 快捷筛选：本月、上月、最近 7 天、最近 30 天。
- 状态筛选。
- 天气筛选。
- 活动备注筛选。
- 缺少洗车数量筛选。
- 区间汇总。
- 数据表格。
- 编辑弹窗或侧边抽屉。
- 删除。
- 操作历史。
- 回退。
- Excel 导出。

表格列根据收入项目动态显示：

```text
日期 | 状态 | 总收入 | 现金 | 刷卡 | 明钱 | 暗钱 | 洗车 | 天气 | 活动 | 记录人 | 最后修改人 | 操作
```

编辑采用弹窗或侧边抽屉，不做表格内直接编辑。

回退规则：

- 修改记录：恢复修改前内容。
- 删除记录：恢复被删除内容。
- 新增记录：回退时删除该新增记录。
- 回退动作也写入审计日志。

Excel 导出包含当前筛选结果和动态收入项目列。

### 6.5 可视化页 `/charts`

第一版只分析当前选中店铺，不做跨店铺图表对比。

顶部筛选：

- 店铺。
- 时间范围：本月、上月、最近 30 天、自定义。
- 收入项目筛选。

KPI：

- 总收入。
- 记录天数。
- 营业天数。
- 主要收入项目金额。
- 有洗车数据时显示总洗车量、平均客单价。

核心图表：

1. 每日收入柱状图。
2. 收入项目构成图。
3. 月度趋势折线图。
4. 天气影响图。
5. 周几表现图。
6. AI 分析当前筛选结果按钮。

收入项目构成图：

- 默认勾选计入总收入的项目。
- 用户可以自行勾选要显示的收入项目。
- 不做“全部项目”一键切换。

洗车数量规则：

- 区间内没有洗车数量数据时，隐藏洗车量和平均客单价相关内容。
- 区间内存在洗车数量数据时，显示完整模式。

第一版不做图片 / PDF 导出，Excel 导出放在数据库页。

### 6.6 工时记录页 `/workers`

该页面是独立模块，不与收入、天气、AI 记账强绑定。

功能：

- 选择店铺。
- 选择月份。
- 管理工人。
- 设置店铺标准工作时长。
- 设置每个工人的满标准日工资。
- 按月填写每天每个工人的工时。
- 支持历史月份修改。
- 支持复制上一天工时。
- 自动计算月度汇总。
- 自动计算工资。
- 生成月度工资结算快照。
- 支持 Excel 导出。

响应式布局：

- 桌面端：月度矩阵表，每行一天，每列一个工人。
- 手机端：按天卡片列表，每张卡片显示当天所有工人。

时间输入：

- 使用滚动选择器。
- `未填写`、`0 小时`、`1 小时` ... `24 小时`。
- 不支持小数小时。

月度汇总：

- 总小时。
- 上班天数。
- 折合标准工作日。
- 满标准日工资。
- 本月工资。
- 本月工人工资合计。

工资规则：

- 标准工作时长按店铺设置。
- 满标准日工资按工人设置。
- 超过标准工作时长时，当天工资封顶。
- 生成结算后保存快照。
- 结算后修改工时，提示需要重新结算。

## 七、天气服务

### 7.1 Open-Meteo

使用 Open-Meteo 作为首选天气服务：

- 预报：`https://api.open-meteo.com/v1/forecast`
- 历史：`https://archive-api.open-meteo.com/v1/archive`
- 地理编码：`https://geocoding-api.open-meteo.com/v1/search`

### 7.2 天气写入策略

记录保存：

- `weather_auto`：API 自动结果。
- `weather`：最终天气，用户可修改，用于展示和分析。
- `weather_code`。
- `temperature_max`。
- `temperature_min`。
- `precipitation`。

规则：

- 用户选择日期时，系统按当前店铺经纬度查询天气。
- 查询失败不阻止记账。
- 普通用户可手动选择天气。
- 天气查不到时可以先空着。
- 后台补偿任务定时重试。
- 用户手动改过 `weather` 后，后台重试只补 `weather_auto` 和基础天气数据，不覆盖用户最终天气。
- 多次重试仍失败，在管理员后台提醒。

## 八、AI Agent 系统

### 8.1 定位

AI 是增强功能，不是基础记账的唯一入口。

AI 采用安全模式：

```text
AI 生成计划，后端校验并执行
```

AI 不直接执行 SQL，不绕过权限，不绕过审批。

### 8.2 能力范围

查询：

- 某天、某周、某月、某区间收入。
- 是否漏记账。
- 最高 / 最低收入日。
- 收入项目金额和占比。

分析：

- 时间趋势。
- 收入项目结构。
- 当前店铺经营总结。
- 天气影响。
- 活动备注影响。
- 周几表现。
- 异常数据检查。

辅助记账：

- 用户可说“今天现金 200，刷卡 150，天气晴，洗了 12 辆”。
- AI 生成草稿。
- 用户确认后保存。
- 信息缺失时草稿不能保存，需要用户补齐。

修改 / 删除：

- 必须生成变更预览。
- 用户确认后后端执行。
- 写入审计日志。

### 8.3 边界规则

- AI 查询和分析可以直接回答。
- AI 新增、修改、删除必须确认。
- AI 只能操作当前选中的店铺。
- AI 不能直接改总收入，只能改收入项目，后端自动重算总收入。
- 普通用户只能分析有权限店铺。
- 管理员可以跨店铺分析。
- AI 使用谨慎表达，不做绝对预测。

### 8.4 对话历史

保存 AI 对话历史，便于用户回看和排查 AI 操作。

## 九、记忆引擎

第一版记忆引擎只做三类规律：

- 天气规律。
- 周几规律。
- 活动备注规律。

原则：

- 每个店铺独立记忆。
- 只做定性描述，不做具体百分比预测。
- 只引用置信度足够高的规律。
- 置信度低于 50 时不主动展示给用户。
- 不把只出现一次的偶然事件包装成规律。
- 第一版不做记忆管理页。

使用场景：

- 首页今日播报。
- 首页明日预告。
- AI 分析。

## 十、定时任务

第一版使用 APScheduler，并放在 FastAPI 后端进程内。后续如果多实例部署，再拆成独立 worker。

### 10.1 每日任务

每个店铺按自己的当地时间凌晨 4 点执行：

1. 查询今日天气。
2. 查询明日天气。
3. 补昨日缺失天气。
4. 扫描新经营数据。
5. 更新记忆引擎。
6. 生成昨日总结。
7. 生成今日播报。
8. 生成明日预告。
9. 检查昨日是否漏记账。
10. 写入任务日志。

### 10.2 补偿任务

后台定期重试：

- 找出缺失天气记录。
- 尝试重新查询天气。
- 成功则补上。
- 失败增加失败次数。
- 多次失败后生成管理员提醒。

补偿任务不影响用户记账。
