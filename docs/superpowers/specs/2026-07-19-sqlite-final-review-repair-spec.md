# AutoLava SQLite 最终审查修复规格

日期：2026-07-19

适用分支：`codex/sqlite-runtime-simplification`
关联设计：`2026-07-18-sqlite-low-memory-runtime-design.md`

## 1. 背景

SQLite 迁移完成后，整分支审查发现以下问题：

1. 历史 Excel 导出使用当前收入分类的名称和顺序标记旧记录，可能误写历史含义。
2. 收入配置和记账请求并发时，记账可能使用等待前读取的旧配置。
3. 管理员 API 的当前态安全行为缺少足够的回归测试。
4. 请求等待天气或 SQLite 写锁期间，账号、权限、门店成员关系或门店状态可能发生变化；旧请求不能继续使用过期授权写入。
5. 数据库读事务不能跨越天气等外部网络调用。

本规格只定义上述最终审查修复，不重新引入审计、回退、配置版本、令牌状态、MySQL 或 Phase 2。

## 2. 修复目标

- 历史记录的导出、页面回填和再次编辑始终使用保存时的项目快照。
- 所有相关 SQLite 写入使用同一个进程级写锁和统一的短事务协议。
- 请求经过网络等待或写锁等待后，必须重新验证当前授权和当前门店状态。
- 天气和其他外部调用期间不得保持数据库事务。
- 管理员高风险操作必须有行为级回归测试。
- 保持既定的最后保存覆盖、整数金额和单 API 进程设计。

## 3. 历史 Excel 导出

### 3.1 工作簿结构

导出文件包含两个工作表：

1. `经营记录`
   - 固定列：日期、状态、总收入、洗车、天气、活动、记录人、最后修改人。
   - 不再根据当前收入分类创建动态金额列。

2. `收入明细`
   - 每条历史收入项目快照占一行。
   - 固定列：日期、收入项目、计入总额、排序、金额。

### 3.2 数据来源

`收入明细` 的以下字段只能来自 `DailyIncomeItem` 保存时的快照：

- `category_name`
- `include_in_total`
- `sort_order`
- `amount`

不得使用当前 `IncomeCategory` 的名称、计入标志或排序覆盖历史含义，也不得只按 `category_id` 合并历史数据。

### 3.3 格式和安全

- 金额写为整数数值，Excel 格式为 `€#,##0`。
- 以 `=`, `+`, `-`, `@` 开头的文本继续进行公式注入转义。

## 4. 统一 SQLite 短写事务

### 4.1 公共协议

所有相关写路径使用统一 helper，执行顺序必须是：

1. 结束依赖或查询阶段打开的旧读事务。
2. 等待全局 `SQLITE_WRITE_LOCK`。
3. 在锁内开启新的数据库读取。
4. 重新验证授权、门店和业务配置。
5. 执行一次短写入。
6. 成功时提交；异常或取消时回滚。
7. 立即释放写锁。

不得在 helper 外复制不同版本的 lock/commit/rollback 协议。

### 4.2 锁内禁止事项

写锁和数据库事务内不得执行：

- 天气、地理编码或其他网络请求；
- 模型/API 请求；
- 睡眠或定时等待；
- 会再次获取同一写锁的嵌套调用。

## 5. 等待后的新鲜授权

### 5.1 不可信旧对象

请求进入路由时由依赖加载的 `User`、`Store` 和 `StoreMember` 只能用于立即读取不可变 ID 或天气所需位置值。

在结束旧事务或等待网络/写锁后，不得继续使用这些 ORM 对象判断授权或业务状态。

### 5.2 锁内重新验证

写入前按 ID 重新加载并验证：

- actor 仍然存在且 `is_active=true`；
- actor 仍具有当前操作需要的 capability；
- 门店仍然存在且处于启用状态；
- 非管理员仍具有当前门店成员关系；
- 当前收入模式和活动分类是最新已提交状态。

状态码保持统一：

- actor 不存在或已停用：401；
- capability 或成员关系被撤销：403；
- 门店不存在或已归档：404。

## 6. 记账流程

### 6.1 天气阶段

1. 从已授权门店复制天气所需的不可变位置数据。
2. 结束依赖打开的读事务。
3. 在无数据库事务状态下执行天气查询。

### 6.2 写入阶段

天气完成后进入统一短写事务，并重新验证 actor、capability、门店和成员关系。

新记录必须在锁内重新读取：

- `Store.income_items_enabled`
- 当前活动收入分类
- 分类名称、是否计入总额和顺序

然后决定 `legacy_total` 或 `composed`，不得使用天气等待前缓存的配置。

已有记录继续保留其原有 `income_mode` 和项目快照。更新和删除同样必须在锁内重新验证当前授权与门店状态。

## 7. 收入配置和管理员操作

以下写操作必须使用统一短写事务并在锁内重新授权：

- 收入配置 replace；
- 分类 archive、restore、永久删除；
- 管理员创建或修改分类；
- 用户创建、修改、停用、降权和删除；
- 门店创建、修改、归档、删除；
- 门店成员替换；
- 其他会改变当前配置或授权状态的管理员写操作。

最后活动管理员和 owner 保护必须根据锁内重新读取的当前数据库状态判断，不能使用等待前计数。

## 8. 分类修改与天气/简报

分类 PATCH 分为两个独立阶段：

1. 第一个短写事务修改当前分类并提交，同时复制门店天气所需的不可变值。
2. 释放锁并结束事务后执行天气请求。
3. 如需刷新简报，进入第二个短写事务，再次验证 actor 和门店后写入简报。

两个阶段不得嵌套获取写锁；天气请求时 session 不得处于事务中。

分类修改不得重新计算或改写往日 `StoreDailyRecord.daily_revenue`，也不得修改往日 `DailyIncomeItem` 快照。

## 9. 管理员回归保障

管理员 API 行为测试至少覆盖：

- 管理员专属授权和普通用户 403；
- owner 与最后活动管理员保护；
- 并发停用或降权不能移除最后管理员；
- 用户名、密码、角色和启用状态验证；
- 门店成员替换及无效门店时的原子性；
- 被真实业务记录引用的用户/门店删除保护；
- 有营业记录的门店归档与无记录门店硬删除；
- 分类归档、恢复、未使用时永久删除及已使用分类保护；
- 相关管理员和配置写入等待共享写锁；
- 等待期间 actor 被停用、降权或移除成员关系后，操作被拒绝且数据库保持不变。

测试不得恢复已经删除的审计、回退、`StoreSetting`、配置版本、令牌状态或 Phase 2 断言。

## 10. 必须具备的交错测试

使用文件型 SQLite 和两个独立 session，确定性覆盖：

1. 记账创建等待天气时 actor 被停用：返回 401，不创建记录。
2. 记账更新等待天气时成员关系被删除：返回 403，旧记录不变。
3. 记账删除等待写锁时门店被归档：返回 404，记录保留。
4. 收入配置和分类操作等待写锁时 actor 被降权：返回 403，配置不变。
5. 用户修改等待写锁时 actor 被停用：返回 401，目标用户不变。
6. 天气 provider 执行时 `session.in_transaction()` 为 false。
7. 配置在记账等待期间从总额模式切换到组合模式：新记录按最新组合模式和最新分类快照保存。

## 11. 验收标准

- Ruff 通过。
- 后端全量测试通过。
- 无 SQLite `ResourceWarning`。
- 历史导出改名/改顺序/修改计入标志回归通过。
- 管理员当前态测试恢复到有意义的行为覆盖，不以行数代替质量。
- 活动源码中不存在 MySQL、审计回退、版本冲突或令牌状态实现。
- 前端 JSON 契约和页面行为未改变时，不要求因 Excel 二进制布局变化重复前端门禁；Excel 使用 `openpyxl` 直接验证。
- Docker-enabled CI、生产烟测和内存快照仍是后续发布条件，不在本修复规格中伪造完成。

## 12. 非目标

- 不恢复 MySQL 数据或本机 MySQL。
- 不新增恢复 API、恢复 UI 或自动恢复。
- 不重新引入审计历史、回退、配置版本或令牌刷新。
- 不实现 Phase 3 Agent、Phase 4 自动化或 `compose.agent.yaml`。
- 不增加多进程 Worker、Redis、队列、内存限制或 24 小时监控。
