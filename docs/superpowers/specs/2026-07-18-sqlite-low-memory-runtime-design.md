# AutoLava SQLite 低内存运行架构设计

**日期：** 2026-07-18

**状态：** 待用户复核

**目标分支：** `test-used`

## 1. 背景与目标

AutoLava 当前运行在 2 核、2 GB 内存的家庭服务器上，实际使用者约为 3～5 人。MySQL 对这一规模的系统占用过多常驻内存，而且服务器后续还需要容纳阶段 3 的 Agent 和阶段 4 的自动化能力。

本次工作的唯一主要目标，是把已经存在的阶段 1 系统从 MySQL 完整迁移为 SQLite，并保持现有业务行为。阶段 3 和阶段 4 本次不实现，只补充未来设计必须遵守的 SQLite、低内存和单实例约束。原阶段 2（员工、工时与工资）不再建设，其计划、路线图入口和产品占位全部删除。

## 2. 已确认的产品决策

- 不迁移现有 MySQL 数据。
- 不为现有 MySQL 数据制作最终备份。
- 切换时直接停止并删除 MySQL 容器及数据卷，然后初始化空 SQLite 数据库。
- 用户自行重新创建管理员、门店、收入项目和少量营业记录。
- SQLite 自动备份只保留最近 3 天。
- 系统面向 3～5 名家庭用户，不以水平扩容和多 API 实例为设计目标。
- 阶段 3 和阶段 4 使用外部模型 API，不在服务器部署本地模型、向量数据库或嵌入服务。
- 阶段 2 整体取消，不保留员工模块入口或未来占位。

## 3. 方案比较

### 方案 A：SQLite 单进程一体化运行（采用）

一份 FastAPI 进程承担业务 API、未来调度器和未来 Agent 请求队列，Nginx 单独提供前端静态资源。SQLite 使用本地持久化卷。

优点是常驻内存最低、部署最简单，适合当前服务器和家庭并发。限制是不能同时运行多个 API 写入实例，未来若需要水平扩容，应迁移至 PostgreSQL 等服务型数据库。

### 方案 B：SQLite 加独立 Agent Worker（不采用）

Agent 在额外进程或容器中执行，可以隔离模型请求，但需要跨进程队列或通信机制，也会增加 SQLite 跨进程写竞争和常驻内存，不适合 2 GB 服务器。

### 方案 C：继续保留并调优 MySQL（不采用）

代码改动较少，并发写入能力更强，但无法实现本次降低常驻内存和简化部署的核心目标。

## 4. 目标运行架构

生产环境只保留两个长期运行的服务：

- `autolava-api`：单实例、单 Uvicorn worker 的 FastAPI 服务；负责业务 API、SQLite 访问，以及未来内嵌的调度器和 Agent 队列。
- `autolava-web`：低内存 Nginx 容器；仅提供预构建前端资源并代理 API。

数据库连接改为 `sqlite+aiosqlite`。数据库文件位于容器内 `/data/autolava.sqlite3`，该目录挂载独立持久化卷，容器更新不得删除数据。

生产服务器不执行前端编译或大型镜像构建。镜像和静态资源应在本地或 CI 中生成。服务器建议配置 2～4 GB swap 作为异常峰值保护，但正常运行不能依赖 swap。

## 5. SQLite 配置与数据一致性

每个数据库连接建立时启用：

- `PRAGMA journal_mode=WAL`
- `PRAGMA foreign_keys=ON`
- `PRAGMA busy_timeout=10000`
- `PRAGMA synchronous=NORMAL`

API 固定为一个进程和一个 worker。所有写操作保持短事务；外部网络请求、文件生成和模型调用不得占用数据库事务。

现有 MySQL 专用行为按以下方式替换：

- `sqlalchemy.dialects.mysql.insert` 改为通用 SQLAlchemy 写法，必要时使用 SQLite `ON CONFLICT DO UPDATE/DO NOTHING`。
- 不依赖 `SELECT ... FOR UPDATE`。优先使用唯一约束、幂等键、条件更新和受影响行数判断；确实需要串行写入的极小范围操作才使用 SQLite 写事务。
- 对可能被重复提交或并发修改的实体增加版本字段或等价的乐观并发检查。冲突时返回明确的业务错误，不静默覆盖新数据。
- 保留金额字段的精确语义。迁移前必须针对 `Numeric` 的写入、读取、汇总和序列化补充 SQLite 回归测试，禁止由浮点转换造成金额误差。
- JSON、时间默认值、级联删除和唯一约束必须通过真实 SQLite 测试验证，不能只依赖模型声明。

预计家庭场景的读请求可以通过 WAL 并行进行；写请求由 SQLite 串行提交。若针对性并发测试出现持续锁库或明显响应恶化，应先缩短事务和减少不必要写入。若业务本身必须支持多个长事务并行写入，则属于 SQLite 兼容效果不佳，必须向用户说明并讨论取舍，不得擅自引入 Redis、独立队列或新数据库服务。

## 6. 空库初始化与 MySQL 删除

本次没有数据迁移和回退数据：

1. 停止旧应用和 MySQL，防止切换过程中继续写入。
2. 删除 MySQL 容器、部署定义、驱动依赖和数据卷。
3. 创建 SQLite 持久化卷并运行 Alembic 迁移。
4. 运行管理员初始化命令。
5. 启动 API 和 Web，执行空库验收。
6. 用户重新创建门店、收入项目和营业记录。

初始化命令必须可重复执行：管理员已存在时不得重复创建或破坏现有账号。数据库或管理员初始化失败时应停止启动并输出明确错误，不能以半初始化状态继续提供服务。

## 7. SQLite 备份

每天执行一次 SQLite 在线备份，只保留最近 3 天：

- 使用 SQLite 在线备份能力生成一致性快照，不能直接复制正在写入的主数据库文件。
- 先写临时文件，完成后运行 `PRAGMA integrity_check`，通过后再原子重命名为正式备份。
- 新备份验证成功后才清理超过 3 天的旧备份。
- 备份失败记录结构化日志并保留之前的有效备份，不阻塞正常业务。
- 数据库文件和备份目录均位于持久化存储，且备份目录设置明确容量边界。
- 恢复只作为服务器管理员操作，不向普通用户提供数据库重置或计划重置入口。

## 8. 现有阶段 1 兼容范围

迁移必须保持现有系统的可见行为，至少覆盖：

- 登录、令牌刷新、权限和管理员初始化。
- 用户、门店、成员关系和门店归档。
- 收入项目配置及其版本行为。
- 每日记账的新建、查看、修改、删除或回滚能力。
- 日历中所有非选中且已有记录的日期显示蓝点。
- 选择已有记录的日期时自动回填数据。
- 成功保存后更新前端表单基准，切换日期不再误报“未保存”。
- 营业记录列表、筛选、统计、图表、审计和任务日志。
- 本地启动器、数据库检查脚本、部署和健康检查。

实现期间逐项审计 MySQL 耦合，包括驱动、连接 URL、测试保护、MySQL 方言 UPSERT、行锁、备份恢复脚本、Compose 服务、部署断言和文档示例。

若某一行为无法在 SQLite 上以相同体验和合理复杂度实现，应暂停该项实现，向用户说明：现有行为、SQLite 限制、可选降级方式及各自代价，由用户决定取舍。

## 9. 阶段规划调整

### 阶段 2

删除员工、工时和工资实施计划，清理路线图、README、本地启动器说明、导航占位和阶段 1 文档中的阶段 2 引用。不得删除阶段 1 已投入使用的业务功能，也不得把删除历史设计文档扩大成无关文档清理。

### 阶段 3：Agent（本次不实现）

后续详细设计时必须重新核对以下约束：

- 模型全部通过外部 API 调用。
- API 内最多同时执行 2 个模型请求，额外请求进入容量为 10 的短队列，最长等待约 120 秒。
- 排队、超时和队列满必须有清晰的前端状态与友好错误。
- 普通用户每日最多主动调用 30 次，管理员不限；系统定时调用不占用户额度。
- 模型请求不得持有 SQLite 事务。
- 涉及业务写入的 Agent 操作仍需审批、权限校验、幂等键和乐观并发检查。

这些数字是当前容量基线，不代表阶段 3 的最终产品设计；阶段 3 开始前仍需单独 brainstorming。

### 阶段 4：自动化与记忆（本次不实现）

后续详细设计时必须重新核对以下约束：

- 调度器在唯一 API 进程内运行，不增加 Redis、Celery、RabbitMQ 或独立调度容器。
- 每个启用门店按门店当地时间每日 04:00 执行固定流程。
- 普通用户不能暂停、修改、重置或手动触发计划。
- 管理员可以手动补跑；同门店不得并行运行，补跑有 5 分钟冷却并复用自动任务的幂等规则。
- 天气补偿每 2 小时运行且不调用模型。
- 每日流程最多调用一次外部模型润色；失败后使用确定性模板，不阻断其他任务。
- API 重启后补查遗漏任务，但不得重复执行已成功任务。

这些规则只记录低内存和数据一致性边界，阶段 4 的功能细节仍需另行讨论。

## 10. 测试与验收

### 针对性验证

- 使用真实临时 SQLite 文件运行后端测试，不使用仅能掩盖方言差异的 mock。
- 验证空库迁移、管理员幂等初始化、外键、级联、唯一约束、JSON、时间和金额精度。
- 为所有替换掉的 MySQL UPSERT 和行锁路径添加并发或重复提交测试。
- 模拟 3～5 个用户的读取和低频写入，确认不会持续出现 `database is locked`。
- 验证每日备份、完整性检查、失败保留旧备份和三天轮换。
- 验证容器重建后数据库与备份仍存在。

### 阶段 1 回归

- 前端全量测试和生产构建。
- 后端全量测试。
- 关键端到端流程：登录、创建门店、配置收入、跨多日期记账、蓝点显示、已有记录回填、修改保存后切换日期、营业记录查询和管理员操作。

### 部署验收

- 生产环境不存在 MySQL 容器、数据卷、驱动或启动依赖。
- 只运行一份 API 和一份 Web。
- 服务器上不执行高内存构建。
- 健康检查验证 API、SQLite 可读写状态和 schema 版本，但不得在每次探活中执行昂贵完整性检查。
- 对内存、磁盘、SQLite 锁等待和备份失败提供可定位日志。

## 11. 非目标

- 不迁移或恢复旧 MySQL 数据。
- 不实现阶段 2。
- 不在本次实现完整阶段 3 Agent 或阶段 4 自动化功能。
- 不部署本地模型、向量数据库、Redis、Celery、RabbitMQ 或独立数据库服务。
- 不支持多 API 实例、水平扩容或高并发写入。
- 不在普通用户界面提供数据库恢复、定时计划修改或重置功能。

## 12. 风险与退出条件

- 直接删除 MySQL 意味着旧数据不可恢复，这是用户明确接受的决策。
- SQLite 适合当前家庭负载，但单写者是明确限制。若后续用户量、自动任务写入量或 Agent 写操作显著增长，应重新评估 PostgreSQL。
- 单 API 进程也意味着调度和 API 共用故障域；现阶段通过幂等补跑、短任务和外部模型超时控制风险。
- 任何需要牺牲现有阶段 1 数据正确性或明显降低用户体验的兼容问题，均为讨论触发点，不能默认接受。

## 13. 完成定义

当以下条件全部满足时，本次迁移才算完成：

1. 阶段 1 全部核心行为在 SQLite 上通过测试和关键端到端验收。
2. MySQL 代码、运行依赖、容器和数据卷已从目标环境删除。
3. SQLite 数据持久化、三天备份和初始化流程经过实际验证。
4. 阶段 2 计划和产品占位已移除。
5. 阶段 3、4 文档明确记录 SQLite 单实例与低内存边界，但没有提前实现其业务功能。
6. 服务器最终只运行预构建的 API 与 Web 服务，内存占用符合 2 GB 主机约束。
