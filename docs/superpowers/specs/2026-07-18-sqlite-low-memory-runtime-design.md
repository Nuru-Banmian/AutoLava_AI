# AutoLava SQLite 精简运行架构设计

**日期：** 2026-07-18

**状态：** 待用户最终复核

**目标分支：** `test-used`

## 1. 背景与目标

AutoLava 当前运行在 2 核、2 GB 内存的家庭服务器上，实际使用者约为 3～5 人。现有 MySQL 对这一规模的系统占用过多常驻内存，服务器后续还可能容纳阶段 3 Agent 和阶段 4 自动化能力。

本次工作的主要目标是把已经存在的阶段 1 系统从 MySQL 完整改为 SQLite，并主动删除家庭场景不需要的历史、回退、配置版本和令牌状态功能。迁移后只运行一份 API 和一份 Web，保持每日记账、往日营业数据查询、日历蓝点、旧日期自动回填、经营分析及当前门店管理能力。

阶段 2（员工、工时和工资）整体取消。阶段 3、4 本次不实现，只在路线图中留下 SQLite、2 GB 内存和可扩展部署的注意事项，未来分别重新讨论。

## 2. 已确认的产品决策

- 不迁移现有 MySQL 数据。
- 不为现有 MySQL 数据制作最终备份。
- 切换时直接停止并删除 MySQL 容器、数据卷和运行依赖。
- 从空 SQLite 数据库重新创建管理员、门店、收入项目和营业记录。
- 营业额和收入项目金额只使用非负整数，不保存小数金额。
- 删除修改历史、操作审计、历史回退、收入配置版本和并发版本冲突功能。
- 保留真正的往日营业记录、查询、筛选、统计、导出和旧日期编辑。
- 删除刷新令牌、令牌记录、服务端登录会话和数据库中的 `remember_token`。
- 使用固定 24 小时有效期的无状态 JWT HttpOnly Cookie；过期后重新登录。
- SQLite 每天自动备份一次，只保留最近 3 个自然日。
- 系统不提供任何恢复页面、API 或恢复脚本；必要时由服务器管理员停机后手工替换数据库文件。
- 不设置 API 或 Web 容器内存上限，不引入长期监控或本阶段压力测试脚本。
- 阶段 2 整体删除；阶段 3、4 以后另行 brainstorming。

## 3. 方案选择

### 3.1 采用：SQLite 单进程一体化运行

一份 FastAPI 进程承担业务 API、现有轻量后台任务和每日数据库备份，Nginx 单独提供预构建前端资源。SQLite 文件放在 API 的持久化卷内。

该方案常驻内存最低、部署简单，适合 3～5 名家庭用户。明确限制是不能运行多个 API 写入实例，也不以高并发写入或水平扩容为目标。

### 3.2 暂不采用：独立 Agent Worker

本次不创建 Worker、任务表、任务队列或空内部接口。未来阶段 3 如果需要独立 Worker，可以通过额外的 `compose.agent.yaml` 加入服务；Worker 是否通过内部 API 领取任务、是否需要消息队列，等阶段 3 再决定。候选原则是独立 Worker 不直接打开 SQLite 文件，避免跨进程数据库写竞争。

### 3.3 不采用：继续保留并调优 MySQL

保留 MySQL 改动较少、并发写入更强，但不能实现降低常驻内存和简化家庭部署的核心目标。

## 4. 目标服务器架构

生产环境只保留两个长期运行的服务：

- `autolava-api`：单实例、单 Uvicorn worker 的 FastAPI 服务，负责业务 API、SQLite 访问、现有天气/简报后台任务和 SQLite 自动备份。
- `autolava-web`：低内存 Nginx 服务，负责预构建静态前端并代理 API。

SQLite 主文件固定放在 `/data/autolava.sqlite3`，备份放在 `/data/backups`，整个 `/data` 目录使用独立 Docker 持久化卷。容器更新不得删除该卷。

生产服务器不进行前端编译或大型镜像构建。API 与 Web 镜像在笔记本电脑或 CI 中构建，服务器只负责加载镜像并使用 `--no-build` 启动。服务器可以配置 2～4 GB swap 作为异常峰值保护，但正常运行不能依赖 swap。

不设置容器内存上限。上线后只记录一次空闲状态和一轮正常完整操作后的 `docker stats`，用于了解第一阶段基础占用；不等待 24 小时，也不在本阶段编写永久压力测试脚本。

## 5. SQLite 连接与写入规则

应用使用 `sqlite+aiosqlite`。每个连接建立时启用：

- `PRAGMA journal_mode=WAL`
- `PRAGMA foreign_keys=ON`
- `PRAGMA busy_timeout=10000`
- `PRAGMA synchronous=NORMAL`

API 固定为一个进程和一个 worker。所有写操作使用短事务；天气请求、文件生成和未来模型调用不得占用数据库事务。

删除所有 MySQL 专用行为：

- 删除 `asyncmy`。
- 删除 `sqlalchemy.dialects.mysql.insert`。
- 管理员初始化和简报写入改用 SQLite `ON CONFLICT DO NOTHING/DO UPDATE`。
- 删除全部 `SELECT ... FOR UPDATE` 和 `with_for_update`。
- 删除针对 MySQL 数据库名称、主机、端口和密码的运行保护。
- 删除 MySQL dump、restore 和数据库客户端调用。

由于旧 MySQL 数据明确不保留，数据库迁移历史直接重新建立：删除现有 `0001`～`0006` MySQL 迁移，创建一份与最终精简模型一致的 SQLite 基线迁移。系统不提供从旧 revision 或旧 MySQL schema 升级、降级的路径。

系统不再解决极少发生的同时编辑覆盖：

- 删除 `row_version` 和前后端 `expected_version`。
- 不再显示“数据已被其他人修改”的冲突提示。
- 两次保存非常接近时，以最后一次成功提交的数据为准。
- 仍保留 `(store_id, date)` 唯一约束，确保同一门店同一日期只有一条记录。
- SQLite 暂时繁忙时等待最多约 10 秒，不增加 Redis 或额外写入队列。

现有后台天气请求可以并行执行，但数据库写回按门店依次完成。后台写入必须在提交前重新读取记录，继续保证用户手工天气不会被自动天气覆盖。

## 6. 整数金额规则

以下金额字段改为整数列：

- `store_daily_records.daily_revenue`
- `daily_income_items.amount`

前端、API schema 和业务服务只接受大于等于 0 的整数，不接受 `1.5`、`1.00` 或其他小数表达。页面和导出显示 `€123`，不再显示 `€123.00`。

分项收入模式下，只对 `include_in_total=true` 的项目求和；未计入总额的项目仍保存和展示，但不影响营业总额。休息日所有金额保存为 `0`。

经纬度、气温和降水量不是金额，继续使用现有小数精度。

## 7. 最终数据模型

新的 SQLite 基线只创建以下主要表：

- `users`
- `stores`
- `store_members`
- `income_categories`
- `store_daily_records`
- `daily_income_items`
- `daily_briefings`
- `scheduled_task_logs`
- `system_alerts`

删除以下表或字段：

- `audit_log`
- `income_config_versions`
- `income_config_version_items`
- `store_settings`
- `users.remember_token`
- `store_daily_records.row_version`
- `store_daily_records.income_config_version_id`
- 所有 Agent、任务队列、刷新令牌和服务端会话预留

`stores` 增加一个简单布尔设置表示当前是否启用分项收入。关闭时直接填写营业总额；开启时填写当前收入项目，并只合计标记为计入总额的项目。

每条每日收入项目继续保存当天的：

- 项目名称
- 是否计入营业总额
- 排列顺序
- 整数金额

这些字段是当日营业记录自身的快照，不属于修改历史。以后重命名、重排或改变当前项目的合计规则，不会重新解释已经保存的旧记录。被旧记录使用的当前收入项目采用归档而不是物理删除，旧记录继续通过自身快照显示。

## 8. 历史、回退与配置版本删除

删除所有审计和回退能力：

- 删除 `AuditLog` 模型、审计服务和回退服务。
- 记账新增、修改、删除和管理员操作不再写审计记录。
- 删除营业记录修改历史 API、回退 API、前端历史列表和确认回退弹窗。
- 删除管理员用户操作历史。
- 删除审计快照保留和清理任务。
- 删除仅为回退竞争、审计唯一性和快照恢复存在的测试。

删除收入配置版本能力：

- 门店只保存当前分项收入开关和当前收入项目。
- 删除版本列表、发布新版本、恢复旧版本接口和测试。
- 新日期使用当前配置；旧日期使用该记录自身的项目快照。

继续保留真正的往日营业数据：

- 营业记录列表、筛选、详情、统计、图表和导出。
- 日历中所有有记录日期的蓝点。
- 选择旧日期后自动回填并允许修改。
- 管理员直接删除某日记录；删除后不可在系统内撤销。

删除用户或门店时仍由真实业务外键决定是否允许物理删除。已经被营业记录引用的用户不能物理删除，只能停用；已有业务数据的门店继续使用归档，不依赖审计表判断。

## 9. 每日记账交互

- 进入每日记账后加载当前选中日期。
- 日期已有记录时，根据该记录的总额模式或项目快照自动填入全部字段。
- 当前可见月份中，所有有数据的非选中日期下方显示蓝点。
- 选中日期使用选中样式，不依赖蓝点表达当前状态。
- 切换月份时查询该月有记录的日期。
- 删除记录后立即移除对应蓝点。
- 第一次保存创建记录，再次保存直接覆盖当前记录。
- 保存成功后立即用服务器返回值重置前端表单基准。
- 保存完成后切换日期不再误报“尚未保存”。
- 用户确实修改了表单但尚未保存时，继续保留离开日期前的提示。

## 10. 登录与认证精简

系统继续使用密码哈希和无状态 JWT，但不保存令牌状态：

- 删除 `remember_token` 字段和所有相关初始化代码。
- 删除登录页“记住我”选项和登录请求的 `remember` 参数。
- 不创建刷新令牌、刷新接口、令牌表或服务端会话表。
- 登录成功后签发固定 24 小时有效的 HttpOnly JWT Cookie。
- JWT 过期或无效时返回登录页，由浏览器密码管理器通过标准 `autocomplete` 保存并填写账号密码。
- 退出登录只清除 Cookie。
- 每次请求仍读取用户并检查 `is_active`，管理员停用用户后立即失效。
- 修改密码不主动失效其他设备已有 Cookie；它们最多在 24 小时后自然过期。

继续保留 Nginx 登录接口限速，但降低内存占用并适应家庭用户：

- 共享限速区从 10 MB 降为 1 MB。
- 每个 IP 平均每分钟允许 10 次登录请求。
- 允许额外突发 10 次。
- 超出后返回 HTTP 429 和友好提示。

## 11. SQLite 自动备份

API 内运行一个轻量定时协程，按服务器配置时区每天凌晨 03:00 创建在线备份：

- 使用 SQLite 在线备份能力生成一致性快照，不直接复制正在写入的数据库文件。
- 先写临时文件，完成后运行 `PRAGMA integrity_check`。
- 检查通过后原子替换当日正式备份。
- 当天重复执行时更新当天文件，不创建大量重复备份。
- API 启动时如果当天尚无有效备份，补做一次；已有当日有效备份则等待下一次 03:00。
- 只保留最近 3 个自然日的有效备份。
- 新备份验证成功后才清理过期文件。
- 备份失败记录日志并保留之前的有效备份，不阻塞记账。

系统不提供恢复功能：

- 不提供恢复页面、API、命令或脚本。
- 不支持单日、单门店或历史版本恢复。
- 真有需要时，由服务器管理员停止 API，手工替换 SQLite 主文件并处理旧的 `-wal`、`-shm` 文件后重启。
- 该操作只写入运维文档注意事项，不实现成应用功能。

现有 `scheduled_task_logs` 只保留最近 7 天；`system_alerts` 中未解决告警继续保留，已解决告警只保留最近 7 天。当前仅用于定位天气、简报和备份运行问题。

## 12. 本地开发与测试

- 执行数据库迁移。
- 幂等初始化管理员。

自动化测试使用真实临时 SQLite 文件，测试结束后清理，不使用 MySQL，也不使用只能掩盖方言差异的数据库 mock。

## 15. 测试与验收

### 15.1 SQLite 与模型

- 新空库可以从唯一 SQLite 基线迁移到最新结构。
- 外键、级联、唯一约束、JSON、时间和整数金额行为使用真实 SQLite 验证。
- 同门店同日期不能创建两条记录。
- 管理员初始化重复运行不会重复创建或覆盖账号。
- SQLite WAL、外键和 10 秒 busy timeout 实际启用。
- 后台天气写入不会覆盖用户手工天气。
- 自动备份可生成、通过完整性检查，并只保留三天。

## 16. 非目标与讨论触发点

本次不迁移旧数据，不实现阶段 2、3、4，不支持多 API 实例、高并发写入或系统内恢复。

如果实施中发现 SQLite 无法在合理复杂度内保持已确认的阶段 1 行为，应暂停对应项并向用户说明：

- 受影响的现有行为。
- SQLite 或精简数据模型造成的限制。
- 可以接受的降级方案。
- 每种方案的内存、复杂度和体验代价。

不得擅自增加 Redis、独立数据库、Worker、消息队列、恢复系统或历史版本功能。
