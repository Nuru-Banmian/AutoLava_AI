# 营业记录分析布局与日期控件设计

## 目标

改善营业记录页桌面端趋势图的可读性，并让桌面与手机端的日期范围控件更容易点击。保持现有日期范围、校验、查询和数据展示逻辑不变。

## 范围

仅调整 `BusinessRecordsPage` 及其直接使用的展示组件：

- 桌面端（`lg` 及以上）将“经营分析”右栏由当前约 `22–24rem` 调整为 `30–32rem`。
- 桌面端趋势图由当前 `18rem` 高调整为 `16rem` 高；其余图表保持既有响应式宽度逻辑。
- 左侧营业记录区使用剩余宽度，不设新的固定宽度；当可用宽度不足时，沿用现有 `lg` 以下的单列布局。
- 页面顶部“记录筛选”的开始日期和结束日期，以及“经营分析”选择“自定义”时的开始日期和结束日期，统一使用更大的原生日期输入框。

不修改接口、日期范围计算、导出、筛选行为、手机端单列布局或已有日历弹出方式。

## 布局行为

### 桌面端

营业记录保持左侧主区，经营分析保持右侧粘性侧栏。右栏目标宽度为 `30–32rem`，从而为曲线图的横轴标签和峰谷趋势增加可读空间。趋势图高度设为 `16rem`，避免右栏在纵向占用过多空间。

### 移动端

不建立双栏；营业记录列表与经营分析继续按当前顺序单列展示。趋势图宽度由容器决定，维持当前响应式行为。

## 日期控件

四个日期输入框使用同一视觉与触控规范：

- 控件整体最小高度为 `44px`。
- 右侧日历触发区域至少 `44px` 宽，包含原生日历图标的可点按空间。
- 保留 `type="date"` 的系统原生日历、键盘输入、最小/最大日期约束与现有无障碍名称。
- 小屏幕上日期输入可换行，但不可产生页面横向滚动，也不可遮挡相邻字段。

不引入自定义日历弹窗，避免改变浏览器原生日历交互和增加维护成本。

## 验收与测试

- 在桌面断点，经营分析栏可获得 `30–32rem` 宽度，趋势图高度为 `16rem`。
- 在移动断点，页面保持单列且没有横向溢出。
- 页面顶部和经营分析自定义范围中的四个日期输入均具备统一的大尺寸点击区域。
- 现有日期范围约束仍有效：开始日期不能晚于结束日期，结束日期不能晚于门店当天日期。
- 更新相关组件测试，覆盖日期输入的可访问名称、尺寸类名/布局契约及现有日期范围请求行为。
